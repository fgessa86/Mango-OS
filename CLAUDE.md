# Mango OS

A single-page BD/pipeline command center for the Saudi healthcare ecosystem. The mental model: every hospital, VC, government body, payer, etc. is an **institution**, and **people** hold roles at institutions. Some institutions are also pipeline **Targets** (sales/BD deals) and/or **Enablers** (partners who can open doors). Built as a personal tool, not a multi-tenant product: no auth, no row-level access control.

## Stack

- React 19 + Vite, plain CSS (`src/styles.css`) with CSS custom properties for theming. No component library, no CSS-in-JS.
- Supabase (Postgres + PostgREST) as the only backend. `src/supabase.js` exports a single `api(table, method, body, query)` helper that hits the REST API directly with the anon key.
- Anthropic API called directly from the browser (`src/anthropic.js`) for AI summaries, photo-note transcription, and transcript summarization. Requires `VITE_ANTHROPIC_API_KEY` at build time (see `.env.example`). There is no backend proxy for this one, so the key is present in the built JS bundle. Acceptable for this project's threat model; revisit if that changes.
- `api/transcribe.js` is the one piece of actual server code in this repo: a Vercel serverless function (auto-detected from the `/api` directory regardless of the Vite framework setting) that proxies voice recordings to OpenAI's Whisper API using a server-only `TRANSCRIBE_API_KEY` env var, since that key must never reach the client. See "Voice transcription" below.
- `src/gmail-sync-updated.js` is a **reference Google Apps Script**, not part of the Vite app: it's pasted into Google Apps Script directly to auto-log Gmail activity into the `activities` table on a timer. It also **auto-ingests Fathom meeting recaps**: emails from `no-reply@fathom.video` with a "Recap for ..." subject are parsed (the full notes live in a hidden `display:none` preview div) into Meeting Purpose / Key Takeaways / Action Items, logged as a structured `meeting` activity against the matched contact (and its linked deal/enabler), with each action item assigned to Fahed becoming a todo. The activity description is prefixed with a `[[FATHOM]]` marker and de-duplicated by title + date. The marker string is a contract shared with the React app (`FATHOM_MARKER` in both files): `ActivityDescription` in `App.jsx` strips it, shows a small purple "Fathom" badge, and renders the structured body (headings, bullets) with light formatting. Any activity whose description looks structured ("Heading:" lines, "- " bullets) gets the same light formatting even without the marker.

- **PWA / app icon**: the mango app icon lives in `public/` as `mango-icon.svg` (master) plus PNGs `mango-icon-180.png` (apple-touch-icon), `mango-icon-192.png` and `mango-icon-512.png` (Android/manifest). `public/manifest.json` (name/short_name "Mango OS", theme_color `#F5A623`, background_color `#FBF8F3`, `display: standalone`, `start_url: /`) makes it installable full-screen from "Add to Home Screen". `index.html` links the manifest, apple-touch-icon, SVG favicon, and a `theme-color` meta (`#FBF8F3`). The PNGs are regenerated from the SVG with `qlmanage -t` + `sips` (no rsvg/ImageMagick in this environment).

Almost all UI logic lives in one file, `src/App.jsx`: the top-level `App` component holds all state and CRUD functions, and every view/sheet/modal is a function component defined below it. `src/constants.js` holds the enum-like arrays (institution types, stages, priorities, cities, etc.) that drive dropdowns and color-coded badges.

## Database schema (Supabase)

The **city** column on deals, enablers, and organizations stores either a plain
string (one city) or a JSON array string like `["Riyadh","Jeddah"]` (many). Use
`parseCities(city)` to read (always returns an array) and `serializeCities(list)`
to write (returns a plain string for one city, a JSON array string for several,
or null for none). Editing an institution's city syncs all its backing rows.

Core entities:
- **deals**: id, company, stage, **tier** ("Tier 1"/"Tier 2"/"Tier 3"/"Untiered", default "Untiered"), contact_id, contact_name, contact_role, value, notes, next_action, city, region, created_at, last_activity_at, ai_summary, ai_summary_updated_at
- **contacts**: id, name, role, company, email, phone, linkedin, source, notes, tags[], warmth, is_internal, **city**, last_contacted_at, created_at, ai_summary, ai_summary_updated_at, **outreach_status** (not_contacted default / awaiting_reply / replied / meeting_booked / nurture), **last_outreach_at**, **awaiting_reply_since** (the Outreach Engine's wait clock), **outreach_channel** (text, nullable: which channel, email / linkedin / phone / whatsapp, the last outreach used; `channelOf`/`suggestChannel` in App.jsx fall back to a best guess from email/linkedin/phone on file when this is unset)
- **enablers**: id, name, type, contact_id, contact_name, notes, city, region, created_at, last_activity_at, ai_summary, ai_summary_updated_at
- **organizations**: id, name, type, sector, description, website, notes, city, region, country (default "Saudi Arabia"), address, **research_data** (JSON string from the "Research Key People" web search), **is_internal** (boolean, default false: flags this institution as our own team, not an external target), created_at, ai_summary, ai_summary_updated_at
- **activities**: id, deal_id, contact_id, enabler_id, organization_id, type, description, created_at, **scheduled_for** (timestamptz, nullable: a future date/time for a `scheduled_meeting` activity, read by the Week in Review report's "Coming Up Next Week" section; not yet written by any UI), **body_snippet** (text, nullable: a cleaned email body excerpt written by the Gmail sync for `email`-type activities), **ai_summary** (text, nullable: a one-sentence AI summary of `body_snippet`, generated on demand and cached here so it never regenerates). Shared activity-timeline log; any subset of the four FKs can be set (an institution activity is logged with all of its backing FKs so it appears everywhere at once). `addActivity` (the single write path, `App.jsx`) auto-resolves and stores the institution FK from the contact's primary role (`contact_roles`) whenever only `contact_id` is given, and rejects an identical activity (same type, description, and contact_id) posted again within 2 minutes, showing "Similar activity just logged" instead of creating a duplicate row.
- **todos**: id, title, description, priority, status, due_date, **deal_id, enabler_id, organization_id, contact_id** (a task can link to any combination of the four; `contact_id` holds the first of possibly several linked people, see **todo_contacts** below), created_at, completed_at.
- **notes**: id, title (default "Untitled"), **content** (text: sanitized HTML from the rich text editor, or legacy plain text from before it existed, see "Rich text notes" below), tags[], **deal_id/enabler_id/organization_id/contact_id** (single-link: at most one is set), **folder_id** (nullable, references note_folders; null = Unfiled), is_pinned, created_at, updated_at. Backs the Notes tab. There is no updated_at trigger, so every write stamps updated_at manually. Note writes patch local `notes` state instead of a full reload, so the editor stays snappy (no cursor jumps while typing).
- **note_folders**: id, name, parent_id (nullable self-reference for nesting), sort_order (default 0), created_at. The folder tree for the Notes tab. Folder writes also patch local `noteFolders` state. Deleting a folder moves its notes to Unfiled (folder_id null) and reparents its direct child folders up to the deleted folder's parent.
- **materials**: id, name, type, audience, version, file_name, **file_data** (base64 data URL), file_size, mime_type, notes, created_at, updated_at. The Materials Library (sales collateral). Files are capped at 5MB and stored inline as base64; because payloads are large, `loadData` selects materials WITHOUT file_data and `downloadMaterial` fetches the body on demand. Type/audience vocabularies are `MATERIAL_TYPES` / `MATERIAL_AUDIENCES` in `App.jsx` (not constants.js, no custom options). No updated_at trigger; writes stamp it manually and patch local state.
- **meeting_briefs**: id, meeting_title, meeting_date, **contact_id/deal_id/enabler_id/organization_id**, brief_content, my_goal, **brief_type** (`external` default or `internal`, see Meeting Prep Briefs), created_at. Saved AI meeting-prep briefs so they can be revisited without regenerating.
- **calendar_events**: id, google_event_id, title, description, location, start_time, end_time, attendees (text[]), attendee_emails (text[]), **matched_contact_id/matched_deal_id/matched_enabler_id/matched_organization_id** (auto-matched by the sync, or set manually via the Calendar's "Link to entity"), **prep_notes** / **outcome_notes** (text, nullable: the event detail panel's Prep Notes and Outcome textareas, see Calendar below), is_past, updated_at, created_at. Populated by the Google Apps Script from Google Calendar (hourly). Backs the Calendar tab and Home's Today's Agenda. The app never writes events (only the matched_* link columns via `linkCalendarEvent`, and prep_notes/outcome_notes via the detail panel); it re-reads them with `refreshCalendar`.
- **email_templates**: id, name, category (intro_request / direct_intro / post_meeting / materials / reengagement / custom), **channel** (text, default "email": email / linkedin / phone / whatsapp), subject, body, created_at, updated_at. Outreach templates (despite the table name, now covers every channel) with merge fields ({first_name}, {full_name}, {institution}, {role}, {my_note}). Eight starter templates (`STARTER_TEMPLATES` in `App.jsx`, five email plus three LinkedIn) are auto-seeded once when the table is empty (never from Boss View). No updated_at trigger; writes stamp it manually and patch local state.
- **boss_comments**: id, author, content, file_name, file_data (base64 data URL), **deal_id/enabler_id/organization_id** (optional entity tag), **is_read** (boolean, default false), created_at. Backs the two-way comment thread between Fahed and Andy: a floating comment button (`CommentPanel`, bottom right, on every tab in both modes) opens a slide-up panel with a 📎 attach button (any file up to 2MB, stored as a base64 data URL in file_data); untagged rows are the general thread, and rows tagged with an entity id also appear in that institution's "Boss Notes" section (`BossNotes`). Fahed's floating button shows an unread badge (comments from Andy newer than a localStorage "last seen" timestamp). Separately, the `is_read` column is the persisted per-comment flag that drives the Home / Command Center "Unread Comments" list and its per-comment "Mark read" button (distinct from the floating badge's timestamp).

Junction / relationship tables:
- **contact_roles**: id, contact_id, entity_type ("deal"/"enabler"/"organization"), entity_id, role_title, is_primary, created_at. The source of truth for who belongs to which institution and in what capacity. A unique index (`uniq_contact_role` on contact_id, entity_type, entity_id, COALESCE(role_title, '')) makes a duplicate role a 409 at the DB level; `persistPersonRole` (App.jsx) checks already-loaded `contactRoles` state first and also catches a 409 from the API itself, treating either as a no-op "Role already exists" rather than an error.
- **deal_contacts** / **enabler_contacts**: legacy per-institution people tables. Every contact_roles write for entity_type "deal"/"enabler" is dual-written here (and vice versa) so the Pipeline deal sheet keeps working. Removing a role cleans up both sides.
- **deal_enablers**: deal_id, enabler_id, relationship, strength, notes (legacy; read-only "Enabler Paths" on the deal sheet).
- **network_edges**: source_type, source_id, target_type, target_id, relationship, strength, direction, notes. Institution-to-institution edges (org-to-org, etc.), person-to-person connections, and (for backward compat) person-to-organization role mirrors. Relationship vocab in `NETWORK_EDGE_RELATIONSHIPS`. **Person-to-person edges are directional and meaningful** (see Person Sheet Connections below): `direction` is `one_way` for asymmetric relationships (`can_introduce`, `reports_to`) or `bidirectional` for symmetric ones (`knows`, `colleague`, `friend`), and for a `can_introduce` edge `source_id` is ALWAYS the introducer and `target_id` the person they can introduce you to. There is no separate `introduced_by` relationship stored: "A was introduced to me by B" is just a `can_introduce` edge with source=B (the introducer), which is why the two people's cards read it in opposite directions.
- **material_links**: id, material_id, **deal_id/enabler_id/organization_id/contact_id**, created_at. Attaches a library material to an entity; the institution sheet looks links up across all of its backing rows and attaches via `institutionPrimaryEntity`.
- **todo_contacts**: id, todo_id, contact_id, created_at, unique(todo_id, contact_id). Lets a task link to any number of people (existing single-contact todos were backfilled in here). `MultiContactPicker` (App.jsx) is the multi-select UI; `syncTodoContacts` reconciles it against a task's full contact list on every save (adds new rows, deletes dropped ones), and `todos.contact_id` is always kept in sync with the first pick for anything that still reads the old single-contact field. `TaskPills` reads `todoContacts` to render one pill per linked person, falling back to `todo.contact_id` for any row with no todo_contacts entry.
- **event_institutions**: id, calendar_event_id, **deal_id/enabler_id/organization_id** (exactly one set per row, like material_links), created_at. Tags a calendar event with an institution the meeting is ABOUT, which may differ from who is on the invite. **event_contacts**: id, calendar_event_id, contact_id, created_at. Tags a calendar event with a person, pre-populated once from attendee_emails matched to an existing contact by email (see Calendar below). Both are annotation tables kept deliberately separate from the synced calendar_events columns, so the hourly Google Calendar re-sync (title/time/attendees) never touches a tag, a prep/outcome note, or a logged outcome.
- **custom_options**: id, field_name, value, created_at. Persisted "+ Add custom" dropdown values, keyed by field_name (see custom-dropdown convention below).

## The institution model (key concept)

An **institution** is a derived object, one per normalized (lowercased/trimmed) name, folding together up to three backing rows: an `organizations` row (its type/city/sector/etc.), a `deals` row (making it a **Target**), and an `enablers` row (making it an **Enabler**). `buildInstitutions(deals, enablers, organizations)` in `App.jsx` produces this list; each institution carries `orgId`/`dealId`/`enablerId`, `isTarget`/`isEnabler`/`isInternal` (from `organizations.is_internal`), resolved `type`/`city`/`stage`, etc. Institution "type" (Hospital, VC, Government, Tech Company, Payer, Regulator, Association, Research, Pharmaceutical, plus custom) is separate from the Target/Enabler/Internal flags. **Internal** marks our own team (e.g. Mango Sciences): a third checkbox beside Target/Enabler that flags the org `is_internal` and is mutually exclusive with Target (checking Internal archives any deal and disables the Target checkbox, since an internal org is never a pipeline target). People linked to an internal institution automatically get `contacts.is_internal = true` (both when the institution is flagged, via `markInstitutionPeopleInternal`, and when a person is later linked, via `addPersonRole` + `entityIsInternal`). The Ecosystem shows internal institutions in a purple "Internal Team" group at the top (pulled out of their type group), with an "Internal only" filter toggle; internal people show a purple "Internal" badge on cards and in the People table; the Network Map renders internal institutions as purple nodes; internal people are one of Path Finder's two root groups (see Network Map below). `institutionTypeMeta`/`normalizeTypeKey` resolve a raw type string to a canonical `{id,label,color}` case-insensitively so legacy free-typed values group correctly.

- **Creating an institution** (`addInstitution`) always creates an `organizations` row (carrying `is_internal` when Internal is checked), and additionally creates a linked `deals` row if Target is checked (skipped when Internal) and/or an `enablers` row if Enabler is checked.
- **Editing an institution** (`saveInstitution`) ensures the org row exists and reconciles the checkboxes: checking Target/Enabler creates the deal/enabler, unchecking archives it via `purgeDeal`/`purgeEnabler` (deletes the deal/enabler and its junctions, but preserves shared activities/todos by nulling their FK). The edit modal confirms before an uncheck removes a linked row.
- **Linking a person to an institution** uses `institutionPrimaryEntity(inst)`: a Target links via its deal (so the person also appears on the Pipeline deal sheet), an Enabler via its enabler, otherwise via its organization. `institutionPeople(inst, ...)` unions people across all three backing rows plus legacy junctions.
- **"+ Add new institution" everywhere an institution is picked**: the Add Person role picker, the Add Role picker, the Task Form's institution `EntityPicker`, the Calendar's "Link to entity" picker, the Quick Add "At institution" cross-link picker, the institution sheet's "Connect Institution" picker, and the Notes "Link to..." picker all end with a "+ Add new institution" option. Picking it swaps the dropdown for an inline `InstitutionCreateForm` (name, type, Target/Enabler checkboxes) right there, no navigation. `createInstitutionInline` (App.jsx) creates the organizations row plus an optional deal/enabler exactly like `addInstitution`, then returns `{orgId, dealId, enablerId, preferred}` (`preferred` follows the deal > enabler > organization precedence, matching `institutionPrimaryEntity`) so the picker can select the new institution immediately and show "Institution created and selected." `InstitutionSelect` (plain `<select>`, used by most pickers) and `EntityPicker` (searchable typeahead, used by Task Form and Calendar) both accept an optional `onCreateInstitution` prop that wires this up; passing nothing keeps them exactly as before. On a Person Sheet, creating one from Quick Add's "At institution" picker also adds a role for that contact at the new institution in the same step.
- Clicking an institution anywhere, **including a deal card in the Pipeline**, opens the one **unified Institution Sheet** (keyed by name via `institutionSheetKey`). There is no separate Deal Sheet component: a pipeline deal is just an institution backed by a deal row, so it gets the same sheet with all features (People, To-Dos, collapsible AI Summary, Notes, Activity Timeline, Quick Add, Paths In, Connected Institutions). `sheetOrigin` ("pipeline"/"network") only controls the Back button label. Clicking a person opens the **Person Sheet** (keyed by contact id via `personSheetId`).

## Feature map

- **Calendar** (`CalendarTab`, nav id `calendar`, sidebar below Home; on mobile reached via the "Open Calendar" action on Home, no bottom tab): the Google Calendar, read from `calendar_events` (populated hourly by the Apps Script; the app never creates events). Two modes toggled in the header: **Agenda** (default) lists upcoming events grouped by day (Today, Tomorrow, then future days) with the last 2 days collapsed under a "Recent" section; each `CalendarEventRow` shows time range, title, location, attendee names, a clickable matched-entity pill (blue deal / gold enabler / green contact / gray org via `eventMatchClass`), a "Prep Brief" button whenever the event is matched to ANY entity (including an internal team contact) or `isInternalMeeting` detects every attendee is internal team (an @mangosciences.com email, or a contact flagged `is_internal`), showing an "Internal meeting" tag when that fallback applies with no FK match at all, and small indicators (`eventIndicators`) next to the title: 🏷️ if it has any tagged institution or person, 📝 if it has prep or outcome notes, ✅ if an outcome has been logged (derived, not stored: `calendarEventIdFromActivity` scans `activities` for one carrying this event's marker). **Week** is a Mon-Sun grid (`startOfWeek`, prev/next/today nav) with events as colored blocks by matched type, the same indicators shown under the block's title. Unmatched, non-internal events show a "Link to entity" `EntityPicker` (deduped institutions + contacts) that calls `linkCalendarEvent` to set the single matched_* FK. Clicking anywhere on an Agenda row or a Week block (both call `onOpenDetail`, not the row's own inline buttons, which stop propagation) opens the **event detail panel** (`EventDetailPanel`, a global overlay keyed by `eventDetailId`: a right-side panel on desktop, a full-screen sheet on mobile via a `max-width:768px` override), working object for that meeting:
  - **Institutions** and **People** sections: tagged entities as removable pills via `MultiContactPicker` (now generic enough to double as an institution multi-select, with an optional `onCreateInstitution` that adds "+ Add new institution" using the same inline `InstitutionCreateForm` as `EntityPicker`), backed by `event_institutions`/`event_contacts` (`tagEventInstitution`/`untagEventInstitution`/`tagEventPerson`/`untagEventPerson`). These are the institutions and people the meeting is ABOUT, which may differ from who is actually on the Google invite; People is pre-populated once, the first time an event is ever opened, with any attendee whose `attendee_emails` matches an existing contact by email, so a later deliberate removal is never silently re-added.
  - **Prep Notes**: an always-editable textarea, autosaves to `calendar_events.prep_notes` on blur (`saveEventPrepNotes`).
  - **Generate Prep Brief** (`generateEventBrief`): builds the brief from the TAGGED institutions and people via `gatherBriefContextForEntities` (a multi-entity sibling of the single-entity `gatherBriefContext` used elsewhere), not just the auto-matched attendee, so it stays accurate even when the meeting is about entities not on the invite; falls back to the old single-match behavior (`prepBriefForEvent`) when nothing has been tagged yet. Opens in the same `BriefModal` as every other brief, auto-picking External/Internal via the existing `detectBriefType`.
  - **Outcome**: an editable textarea, autosaves to `outcome_notes` on blur, plus a mic button that runs the same Web Speech API + `digestVoiceNote` pattern as `QuickAdd`'s voice notes (the clean summary is appended to the textarea and saved immediately, so "Log Outcome" afterward is the one action that fans it out). "Extract tasks from outcome" reuses `digestVoiceNote` on the typed text to pull `{title, priority, due_date_hint}` action items and creates one todo per item via `saveTodo`, linked to every tagged person (`contact_ids`) and the first tagged institution. **"Log Outcome"** (`logEventOutcome`) is the fan-out: one activity per tagged person (contact_id plus their resolved primary institution FK, the same rule `addActivity` uses elsewhere) and one direct activity for any tagged institution not already reached through a tagged person, every row typed `meeting`, dated to the event's own `start_time`, and prefixed with a `[[CALEVENT:<event id>]]` marker (`CALEVENT_MARKER_PREFIX`/`stripCalendarEventMarker`/`calendarEventIdFromActivity` in `utils.js`, the same pattern as the Fathom recap marker; `cleanActivityText` strips either before display or before feeding an AI prompt). This is what dedupes the same meeting rather than one row per (person x institution) pair; a second "Log Outcome" click for identical text is rejected (exact description match, since these rows are dated to the meeting, not "now", so the usual `addActivity` 2-minute recency check does not apply) with "This outcome is already logged". A toast confirms "Outcome logged to N people and M institutions."
  - **Tagged entity timelines**: since the fan-out relies on `institutionPeople`/cumulative-timeline matching, both the tagged people's and institutions' timelines pick it up automatically; every activity row's `ActivityEntityPills` also shows a "📅 Meeting" pill (only when it carries the marker and `onOpenCalendarEvent` is wired) that reopens this same event detail panel from anywhere (Home's Recent Activity, an Institution Sheet, or a Person Sheet).
  A Refresh button re-reads calendar_events plus event_institutions/event_contacts via `refreshCalendar`, and a "Last synced" time shows `max(updated_at)`. Hidden edit affordances in Boss View (read-only): tags/notes/outcome show as static text, no Log Outcome or Generate Prep Brief. Home's Today's Agenda reads the same `calendar_events` and its meeting cards also open the same detail panel on click (the entity pill and Prep Brief button stop propagation to route to their own actions instead).
- **Home / Command Center** (`HomeTab`, nav id `home`, the default view and the mobile landing tab): a morning briefing. Header greeting (morning/afternoon/evening by clock, "Fahed" or "Andy" in Boss View) plus today's date. Sections, all derived in `App` and passed in: (1) **Unread Comments**, boss_comments from the other person where `is_read = false`, each with avatar/text/timestamp, a clickable entity tag, and a "Mark read" button (`markCommentRead` PATCHes `is_read`); (2) **Today's Agenda**, calendar_events whose start_time is today (time, title, location, matched entity, and a "Prep Brief" button when matched), plus an "Open Calendar" link, else "No events on your calendar today."; (3) **Daily Briefing** (see below); (4) **Briefs**, the 5 most recent meeting_briefs plus "+ New Brief"; (5) **Urgent Tasks**, top 5 open todos ranked overdue then due-today then high-priority (left border red/amber), each with a completion checkbox (hidden in Boss View) and clickable only when it links to an entity; (6) **Recent Activity**, last 10 activities with glyph, description, entity name, timestamp; (7) **Stale Deals**, active deals with no activity in 14+ days. `entityName(row)`/`openEntity(row)` resolve any deal_id/enabler_id/organization_id/contact_id to a name and a navigation target. On mobile, "View Materials" and "View Reports" buttons sit at the bottom (neither has a mobile tab).
- **Daily Briefing** (`DailyBriefing` in `App.jsx`, on Home below Today's Agenda): three collapsible news sections (Health Tech and AI, Oncology and Immunotherapy, Saudi Arabia), each fetched by `fetchNewsStories` in `anthropic.js` (claude-sonnet-4-6 + web_search tool, max_tokens 1500, 90s timeout, JSON-array-only response `[{headline, summary, source, url}]` parsed via `parseJsonArray`). Each story row is a clickable `<a target="_blank" rel="noopener noreferrer">` that opens the source article. Article URLs are resolved deliberately (the model tends to return a publication homepage instead of the story): the prompt demands the DIRECT article link (or null); `fetchNewsStories` uses `webSearchTextWithSources` to also harvest the real result URLs from the `web_search_tool_result` blocks (and text-block citations), and `pickArticleUrl` keeps the model's url only when it is a deep link, otherwise matches the story to a result URL by headline/title word overlap plus a source-domain bonus. `isHomepageUrl` flags a bare-domain URL (no path); anything still resolving to a homepage falls back to a Google search link for the headline. `newsStoryHref` re-applies that homepage guard at render time (protecting older cached payloads). The three fetches are **staggered by 1s** (fire, +1s, +2s) so simultaneous web searches do not rate-limit each other, and each is **resilient**: `fetchSection` retries up to 2 times with a 2s backoff, then falls back to `fetchNewsStoriesNoSearch` (a plain, no-web-search call on the model's own knowledge using the section's `fallbackPrompt`) before surfacing an error. Every attempt's real error is `console.warn`/`console.error`-logged (`[News] <section> attempt n/3 failed: ...`). `parseJsonArray` also does **partial recovery**: if a response is truncated by max_tokens (no closing `]`), `recoverJsonObjects` salvages every complete `{...}` object rather than failing. A failed section shows "Couldn't load [section] news. Tap to retry." (the button shows "Retrying..." and re-attempts just that one section) without touching the others; loading shows shimmer skeletons. A "Summarize" link appears while a snippet exists but `ai_summary` does not; clicking it sends the snippet to Claude (`generateSummary`, max_tokens 150, one-sentence prompt), PATCHes the result onto `activities.ai_summary`, and from then on the summary (with a small purple AI badge) replaces the raw snippet, no repeat API calls. Every activity row also shows the exact timestamp (`formatDateTime`, e.g. "Jul 15, 9:42 AM") rather than just a relative date.
- **Global search** (`GlobalSearch`): the one universal search experience, reachable identically from Cmd+K / Ctrl+K, the sidebar Search button, the magnifier on Home (the mobile entry point, since the sidebar is hidden on phones), and the Ecosystem tab's "Search people and institutions..." button. A client-side overlay over already-loaded data (no per-keystroke API calls), grouped results (5 per group, 2+ characters to search) across institutions (matched by name, type, or city), people (matched by name, role, email, or their institution, resolved via `resolveContactRoles` when it isn't on the raw company field), activities, tasks, notes, and materials; click navigates to the right sheet, Enter opens the top result, Esc closes.
- **Theme**: light only (the warm cream design system). Dark mode has been removed.
- **Warmth**: per-contact relationship strength, a colored dot on people cards and person sheets, and the basis for Path Finder's roots (see Network Map above). Meanings: **active** (a direct, ongoing relationship, already in contact, the strongest tier and a Path Finder root), **hot** (a strong relationship, highly responsive, but not necessarily active right now), **warm** (know them, or a connection warm enough that they would take a call or make an introduction), **cold** (identified but no real relationship yet), **unknown** (not assessed).

## Conventions worth knowing before editing

- **Custom dropdowns**: every categorical dropdown uses `SelectWithCustom` (drop-in `<select>` that appends "+ Add custom..."; picking it swaps to a text input). To persist custom values, pass `options={optionsWithCustom(DEFAULTS, customOptions, "field_name")}` and wrap `onChange` with `trackCustom("field_name", thatSameOptionsList, onAddCustomOption)`. `customOptions` (raw rows) and `onAddCustomOption` (`App`'s `addCustomOption`) are threaded down as props. Field names in use: institution_type, city, region, deal_stage, warmth, relationship, activity_type, priority, tag, strength, enabler_type. Multi-select tags use `TagPickerWithCustom`; colored button groups (warmth) use `ButtonGroupWithCustom`. All live near the top of `App.jsx`.
- Every "delete" CRUD function manually cascades related rows first, since there is no guaranteed `ON DELETE CASCADE`. The cascade covers ALL dependent tables: junctions (deal_contacts/enabler_contacts/contact_roles/deal_enablers), network_edges (both directions), material_links (deleted), and history rows (activities, todos, notes, meeting_briefs, boss_comments, calendar_events' matched_* links) which are PATCHed to null the dangling reference so their content survives. `deleteContact` also strips the denormalized contact_id/contact_name/contact_role copies on deals and enablers. `deleteFullyOrphanedActivities` sweeps activities whose four entity FKs are all null. When adding a new table with entity FKs, extend deleteContact, purgeDeal, purgeEnabler, and deleteInstitution.
- **Targeted state updates**: high-frequency writes (inline field saves, stage/tier moves, todo add/toggle/update, activity logging, outreach recording, comments, custom options, AI summaries) PATCH/POST and then patch local React state; they must NOT call `loadData()`. Structural operations (create/delete entities, role changes, research) still do a full reload. A task checkbox toggle is exactly 1 HTTP request. The "Refresh data" button (sidebar) / refresh icon (Home) is the manual way to pull external changes (Gmail sync, Boss View edits).
- `persistContact`'s company auto-link dual-writes contact_roles (source of truth) plus the legacy junction row; keep any new people-linking write path doing the same (see `persistPersonRole`).
- **In-flight guard on every create action**: every button that POSTs (add role, add person, add institution, add activity, add task, add connection, add note, attach material, add comment) tracks its own `saving`/`posting`/`adding` boolean, checks it at the top of the handler (`if (... || saving) return;`), sets it before the request and clears it in a `finally`, and disables the button while true. This is what stops a double-click (or an accidental double-fire) from creating two rows; when adding a new create handler, follow this pattern rather than relying on the API alone. `api()` (`supabase.js`) attaches the HTTP status to a thrown error's `.status` so a 409 (unique constraint, e.g. `contact_roles`) can be told apart from a real failure and treated as a graceful no-op.
- Colors for enum-like fields are defined once in `constants.js` as `{ id, label, color }` and rendered as `<span className="badge" style={{background: color+"22", color, border: \`1px solid ${color}44\`}}>`. Reuse this rather than inventing badge styles.
- Location fields use `SelectWithCustom` against `CITY_OPTIONS`/`REGION_OPTIONS`. Render a city as a small pin, `📍 {city}`, with the `.city-pin` class.
- Only send optional string fields to Supabase when they have real content (trim and skip empties); never send an empty string for a numeric column.
- Never use the em dash character anywhere: not in UI text, labels, toasts, descriptions, generated content, AI prompts, or code comments. Use commas, periods, colons, or parentheses instead. The AI prompts explicitly instruct the model to avoid them too.
- No test suite exists. Verify changes by running the dev server and exercising the feature directly. This app talks to a live, shared Supabase project with real data, so clean up any test data afterward, and do not assume a row is test data just because it appeared recently; verify before deleting.

## Mobile responsiveness

The app is responsive at `max-width: 768px` (phones); most of the mobile layout lives in one media block at the bottom of `src/styles.css`. Key behaviors: the left sidebar is hidden and replaced by a fixed bottom **tab bar** (`MobileTabBar` in `App.jsx`, rendered next to `CommentPanel`; **Home/Pipeline/Ecosystem/Tasks/Outreach**, Outreach hidden in Boss View, active tab in mango gold, `env(safe-area-inset-bottom)` for notches). Global search is reached from the magnifier on Home. The Network Map is **desktop-only**: it is dropped from the mobile tab bar, and if `view === "map"` on a phone the map renders a "Network Map is best viewed on desktop." notice (`.map-mobile-notice`) instead of the D3 graph. Reports, Notes, and Materials have no mobile tab either; Reports and Materials are reached from "View Reports" / "View Materials" buttons at the bottom of Home, and Notes from a "My Notes" section on Home (the Notes list and editor are separate full-screen views on mobile, the editor gaining a back button). The Daily Briefing news sections start collapsed on mobile. The floating comment button sits just above the tab bar, right-aligned (`right: 16px; bottom: calc(56px + env(safe-area-inset-bottom) + 12px)`), with the Ecosystem add FAB stacked above it; `.main` gets extra bottom padding so content scrolls clear of both. These two floating-button overrides live in a final media block at the very end of `styles.css` because the base `.comment-fab`/`.comment-panel` rules are defined after the main responsive block and would otherwise win the cascade. The Pipeline shows one full-width stage column at a time with CSS scroll-snap; `MobilePipelineNav` (a header with prev/next arrows over a `kanbanRef`) tracks and scrolls the centered column. The Ecosystem "+ Institution"/"+ Person" buttons collapse into a floating add button (`.mobile-add-fab` in `NetworkTab`); People forces card view (`useIsMobile()` -> `effectivePeopleView`) with a "Switch to desktop for table editing" note. Sheets/modals go full width (modals become full-screen sheets, inputs 16px to avoid iOS zoom), stats become a 2x2 grid, tap targets are >= 44px, and the Network Map side panel becomes a 60% bottom sheet with larger person node tap targets (min 16px radius) and `touch-action: none` for pinch/pan. The floating comment button moves above the tab bar (bottom-left, clear of the add FAB). `useIsMobile(768)` is a small `matchMedia` hook for the few JS-driven switches CSS cannot do.

## Voice transcription (`VoiceRecorder`, `api/transcribe.js`)

Audio dictation is real audio, not the browser's Web Speech API (which iOS Safari and most non-Chrome mobile browsers never supported at all). `src/VoiceRecorder.jsx` is the one recorder component used everywhere: it calls `navigator.mediaDevices.getUserMedia({ audio: true })` and records with `MediaRecorder`, picking whichever mime type the browser actually supports (`audio/webm;codecs=opus` on Chrome/Firefox, `audio/mp4` on iOS Safari; `pickSupportedMimeType()` tries a candidate list via `MediaRecorder.isTypeSupported` and falls back to the browser default if none match). On stop, the recorded `Blob` is base64-encoded and POSTed as JSON (`{ audio, mimeType }`) to `/api/transcribe`, a Vercel serverless function (`api/transcribe.js`) that wraps it in multipart `FormData` and calls OpenAI's Whisper API (`whisper-1`) using a server-only `TRANSCRIBE_API_KEY` env var (never exposed to the client, unlike the `VITE_ANTHROPIC_API_KEY` pattern used elsewhere in this app). The function returns `{ text }` on success or `{ error }` (with a specific message) on failure; `maxDuration: 60` is set in the function's exported `config`, and the practical recording length is bounded by Vercel's platform-wide 4.5MB request body cap (roughly a couple of minutes of compressed audio) since serverless functions outside Next.js have no configurable body size limit.

`VoiceRecorder` shows a round mic button (▸ 🎙, red and pulsing while recording with a running mm:ss timer, ▸ ⏹ to stop), "Transcribing..." while uploading, a specific error message with **Retry** (re-uploads the same in-memory blob, discarded once transcription succeeds either way) and **Dismiss** on failure, and "Microphone access denied. Enable it in your browser settings." on a permission rejection. Two modes: `mode="plain"` hands the raw transcript straight to `onPlainText(text)` for dictation into a field (the caller appends it); `mode="digest"` additionally runs the transcript through the existing `digestVoiceNote()` (`src/anthropic.js`, unchanged) for a clean summary plus action items, handed to `onDigest({summary, action_items})`. A `compact` prop shrinks the button (32px vs 40px) for inline placements next to a text field rather than a standalone action.

Placements: **Quick Add** (`mode="digest"`, same "Meeting Note" card + Save Note / Save Note + Create Tasks flow as before, now fed by real transcription instead of speech recognition); **Calendar event Outcome** (`mode="digest"`, appends the summary into the Outcome textarea and saves immediately); **Calendar event Prep Notes** (`mode="plain"`, compact, appends dictated text); **Notes tab body** (`mode="plain"`, appends a new paragraph into the rich editor); **entity sheet Notes/Description scratchpads** (`mode="plain"`, compact); **Task title** (`mode="plain"`, compact, appends to the title input). `showToast` is threaded down from whichever ancestor already has it (`App`, `NotesTab` -> `NoteEditor`, `TaskForm`) for the digest-mode failure toast; it is optional and a safe no-op where a caller has no toast handy (`mode="plain"` never needs it).

## Rich text notes (`RichTextEditor`, `richtext.js`)

The Notes tab body, and the lighter entity-sheet scratchpads (a contact's Notes, an institution's Description and Notes), store **sanitized HTML** in their existing `text` columns (`notes.content`, `contacts.notes`, `organizations.notes`, `organizations.description`) rather than markdown or plain text: no schema change, and a contentEditable surface naturally produces HTML. `src/RichTextEditor.jsx` exports the editor (a contentEditable div plus a toolbar, formatting via `document.execCommand`: bold/italic/underline, H1-H3 via `formatBlock`, bulleted/numbered lists, a checklist button that inserts `<div class="rt-check"><input type="checkbox" contenteditable="false"></div>`, a link button (`window.prompt` for the URL), and a blockquote) and `RichTextView` (a read-only `dangerouslySetInnerHTML` renderer for Boss View and other static contexts, where checklist checkboxes are made inert with `pointer-events: none` in CSS rather than a `disabled` attribute, since `disabled` would also freeze them inside the live editor). A `mini` prop drops the toolbar down to just Bold and Bulleted list for the entity scratchpads (Part 2e's "keep it simple, but at minimum line breaks, bold, and bullet lists"); the full toolbar is only on the Notes tab. Cmd/Ctrl+B/I/U are wired as keyboard shortcuts.

`src/richtext.js` holds the shared logic: `sanitizeHtml(html)` (an allowlist walk over a detached `DOMParser` tree, stripping any tag/attribute not on the list, forcing `target="_blank" rel="noopener noreferrer"` on links and dropping `javascript:` hrefs) runs on every save, since a paste can carry arbitrary markup along from the clipboard; `looksLikeHtml`/`plainTextToHtml` distinguish and migrate a legacy plain-text value (written before this editor existed) into the same HTML shape on load, so old notes keep rendering correctly (Part 2f) without a migration; `toDisplayHtml` is what both the editor and `RichTextView` actually render; `stripHtmlToText` powers list previews and linked-entity snippets (Part 2c: formatting never leaks into a snippet). Checkbox toggling inside the editor is a `contenteditable="false"` input (so a click toggles it rather than placing a text cursor) with a manual `click` handler that syncs the DOM `checked` attribute to the live `.checked` property, since the attribute (what actually serializes into saved HTML) does not update on its own when a user clicks a checkbox.

# Mango OS

A single-page BD/pipeline command center for the Saudi healthcare ecosystem. The mental model: every hospital, VC, government body, payer, etc. is an **institution**, and **people** hold roles at institutions. Some institutions are also pipeline **Targets** (sales/BD deals) and/or **Enablers** (partners who can open doors). Built as a personal tool, not a multi-tenant product: no auth, no row-level access control.

## Stack

- React 19 + Vite, plain CSS (`src/styles.css`) with CSS custom properties for theming. No component library, no CSS-in-JS.
- Supabase (Postgres + PostgREST) as the only backend. `src/supabase.js` exports a single `api(table, method, body, query)` helper that hits the REST API directly with the anon key; there is no server code in this repo.
- Anthropic API called directly from the browser (`src/anthropic.js`) for AI summaries, photo-note transcription, and transcript summarization. Requires `VITE_ANTHROPIC_API_KEY` at build time (see `.env.example`). There is no backend proxy, so the key is present in the built JS bundle. Acceptable for this project's threat model; revisit if that changes.
- `src/gmail-sync-updated.js` is a **reference Google Apps Script**, not part of the Vite app: it's pasted into Google Apps Script directly to auto-log Gmail activity into the `activities` table on a timer. It also **auto-ingests Fathom meeting recaps**: emails from `no-reply@fathom.video` with a "Recap for ..." subject are parsed (the full notes live in a hidden `display:none` preview div) into Meeting Purpose / Key Takeaways / Action Items, logged as a structured `meeting` activity against the matched contact (and its linked deal/enabler), with each action item assigned to Fahed becoming a todo. The activity description is prefixed with a `[[FATHOM]]` marker and de-duplicated by title + date. The marker string is a contract shared with the React app (`FATHOM_MARKER` in both files): `ActivityDescription` in `App.jsx` strips it, shows a small purple "Fathom" badge, and renders the structured body (headings, bullets) with light formatting. Any activity whose description looks structured ("Heading:" lines, "- " bullets) gets the same light formatting even without the marker.

- **PWA / app icon**: the mango app icon lives in `public/` as `mango-icon.svg` (master) plus PNGs `mango-icon-180.png` (apple-touch-icon), `mango-icon-192.png` and `mango-icon-512.png` (Android/manifest). `public/manifest.json` (name/short_name "Mango OS", theme_color `#F5A623`, background_color `#FBF8F3`, `display: standalone`, `start_url: /`) makes it installable full-screen from "Add to Home Screen". `index.html` links the manifest, apple-touch-icon, SVG favicon, and a `theme-color` meta (`#FBF8F3`). The PNGs are regenerated from the SVG with `qlmanage -t` + `sips` (no rsvg/ImageMagick in this environment).

Almost all UI logic lives in one file, `src/App.jsx`: the top-level `App` component holds all state and CRUD functions, and every view/sheet/modal is a function component defined below it. `src/constants.js` holds the enum-like arrays (institution types, stages, priorities, cities, etc.) that drive dropdowns and color-coded badges.

## Database schema (Supabase)

The **city** column on deals, enablers, and organizations stores either a plain
string (one city) or a JSON array string like `["Riyadh","Jeddah"]` (many). Use
`parseCities(city)` to read (always returns an array) and `serializeCities(list)`
to write (returns a plain string for one city, a JSON array string for several,
or null for none). Editing an institution's city syncs all its backing rows.

Core entities:
- **deals**: id, company, stage, **tier** ("Tier 1"/"Tier 2"/"Tier 3"/"Untiered", default "Untiered"), contact_id, contact_name, contact_role, value, notes, next_action, city, region, created_at, last_activity_at, ai_summary, ai_summary_updated_at
- **contacts**: id, name, role, company, email, phone, linkedin, source, notes, tags[], warmth, is_internal, **city**, last_contacted_at, created_at, ai_summary, ai_summary_updated_at, **outreach_status** (not_contacted default / awaiting_reply / replied / meeting_booked / nurture), **last_outreach_at**, **awaiting_reply_since** (the Outreach Engine's wait clock)
- **enablers**: id, name, type, contact_id, contact_name, notes, city, region, created_at, last_activity_at, ai_summary, ai_summary_updated_at
- **organizations**: id, name, type, sector, description, website, notes, city, region, country (default "Saudi Arabia"), address, **research_data** (JSON string from the "Research Key People" web search), **is_internal** (boolean, default false: flags this institution as our own team, not an external target), created_at, ai_summary, ai_summary_updated_at
- **activities**: id, deal_id, contact_id, enabler_id, organization_id, type, description, created_at. Shared activity-timeline log; any subset of the four FKs can be set (an institution activity is logged with all of its backing FKs so it appears everywhere at once). `addActivity` (the single write path, `App.jsx`) auto-resolves and stores the institution FK from the contact's primary role (`contact_roles`) whenever only `contact_id` is given, and rejects an identical activity (same type, description, and contact_id) posted again within 2 minutes, showing "Similar activity just logged" instead of creating a duplicate row.
- **todos**: id, title, description, priority, status, due_date, **deal_id, enabler_id, organization_id, contact_id** (a task can link to any combination of the four), created_at, completed_at.
- **notes**: id, title (default "Untitled"), content, tags[], **deal_id/enabler_id/organization_id/contact_id** (single-link: at most one is set), **folder_id** (nullable, references note_folders; null = Unfiled), is_pinned, created_at, updated_at. Backs the Notes tab. There is no updated_at trigger, so every write stamps updated_at manually. Note writes patch local `notes` state instead of a full reload, so the editor stays snappy (no cursor jumps while typing).
- **note_folders**: id, name, parent_id (nullable self-reference for nesting), sort_order (default 0), created_at. The folder tree for the Notes tab. Folder writes also patch local `noteFolders` state. Deleting a folder moves its notes to Unfiled (folder_id null) and reparents its direct child folders up to the deleted folder's parent.
- **materials**: id, name, type, audience, version, file_name, **file_data** (base64 data URL), file_size, mime_type, notes, created_at, updated_at. The Materials Library (sales collateral). Files are capped at 5MB and stored inline as base64; because payloads are large, `loadData` selects materials WITHOUT file_data and `downloadMaterial` fetches the body on demand. Type/audience vocabularies are `MATERIAL_TYPES` / `MATERIAL_AUDIENCES` in `App.jsx` (not constants.js, no custom options). No updated_at trigger; writes stamp it manually and patch local state.
- **meeting_briefs**: id, meeting_title, meeting_date, **contact_id/deal_id/enabler_id/organization_id**, brief_content, my_goal, **brief_type** (`external` default or `internal`, see Meeting Prep Briefs), created_at. Saved AI meeting-prep briefs so they can be revisited without regenerating.
- **calendar_events**: id, google_event_id, title, description, location, start_time, end_time, attendees (text[]), attendee_emails (text[]), **matched_contact_id/matched_deal_id/matched_enabler_id/matched_organization_id** (auto-matched by the sync, or set manually via the Calendar's "Link to entity"), is_past, updated_at, created_at. Populated by the Google Apps Script from Google Calendar (hourly). Backs the Calendar tab and Home's Today's Agenda. The app never writes events (only the matched_* link columns via `linkCalendarEvent`); it re-reads them with `refreshCalendar`.
- **email_templates**: id, name, category (intro_request / direct_intro / post_meeting / materials / reengagement / custom), subject, body, created_at, updated_at. Outreach email templates with merge fields ({first_name}, {full_name}, {institution}, {role}, {my_note}). Five starter templates (`STARTER_TEMPLATES` in `App.jsx`) are auto-seeded once when the table is empty (never from Boss View). No updated_at trigger; writes stamp it manually and patch local state.
- **boss_comments**: id, author, content, file_name, file_data (base64 data URL), **deal_id/enabler_id/organization_id** (optional entity tag), **is_read** (boolean, default false), created_at. Backs the two-way comment thread between Fahed and Andy: a floating comment button (`CommentPanel`, bottom right, on every tab in both modes) opens a slide-up panel with a 📎 attach button (any file up to 2MB, stored as a base64 data URL in file_data); untagged rows are the general thread, and rows tagged with an entity id also appear in that institution's "Boss Notes" section (`BossNotes`). Fahed's floating button shows an unread badge (comments from Andy newer than a localStorage "last seen" timestamp). Separately, the `is_read` column is the persisted per-comment flag that drives the Home / Command Center "Unread Comments" list and its per-comment "Mark read" button (distinct from the floating badge's timestamp).

Junction / relationship tables:
- **contact_roles**: id, contact_id, entity_type ("deal"/"enabler"/"organization"), entity_id, role_title, is_primary, created_at. The source of truth for who belongs to which institution and in what capacity.
- **deal_contacts** / **enabler_contacts**: legacy per-institution people tables. Every contact_roles write for entity_type "deal"/"enabler" is dual-written here (and vice versa) so the Pipeline deal sheet keeps working. Removing a role cleans up both sides.
- **deal_enablers**: deal_id, enabler_id, relationship, strength, notes (legacy; read-only "Enabler Paths" on the deal sheet).
- **network_edges**: source_type, source_id, target_type, target_id, relationship, strength, direction, notes. Institution-to-institution edges (org-to-org, etc.), person-to-person connections, and (for backward compat) person-to-organization role mirrors. Relationship vocab in `NETWORK_EDGE_RELATIONSHIPS`.
- **material_links**: id, material_id, **deal_id/enabler_id/organization_id/contact_id**, created_at. Attaches a library material to an entity; the institution sheet looks links up across all of its backing rows and attaches via `institutionPrimaryEntity`.
- **custom_options**: id, field_name, value, created_at. Persisted "+ Add custom" dropdown values, keyed by field_name (see custom-dropdown convention below).

## The institution model (key concept)

An **institution** is a derived object, one per normalized (lowercased/trimmed) name, folding together up to three backing rows: an `organizations` row (its type/city/sector/etc.), a `deals` row (making it a **Target**), and an `enablers` row (making it an **Enabler**). `buildInstitutions(deals, enablers, organizations)` in `App.jsx` produces this list; each institution carries `orgId`/`dealId`/`enablerId`, `isTarget`/`isEnabler`/`isInternal` (from `organizations.is_internal`), resolved `type`/`city`/`stage`, etc. Institution "type" (Hospital, VC, Government, Tech Company, Payer, Regulator, Association, Research, Pharmaceutical, plus custom) is separate from the Target/Enabler/Internal flags. **Internal** marks our own team (e.g. Mango Sciences): a third checkbox beside Target/Enabler that flags the org `is_internal` and is mutually exclusive with Target (checking Internal archives any deal and disables the Target checkbox, since an internal org is never a pipeline target). People linked to an internal institution automatically get `contacts.is_internal = true` (both when the institution is flagged, via `markInstitutionPeopleInternal`, and when a person is later linked, via `addPersonRole` + `entityIsInternal`). The Ecosystem shows internal institutions in a purple "Internal Team" group at the top (pulled out of their type group), with an "Internal only" filter toggle; internal people show a purple "Internal" badge on cards and in the People table; the Network Map renders internal institutions as purple nodes and roots Path Finder at internal people. `institutionTypeMeta`/`normalizeTypeKey` resolve a raw type string to a canonical `{id,label,color}` case-insensitively so legacy free-typed values group correctly.

- **Creating an institution** (`addInstitution`) always creates an `organizations` row (carrying `is_internal` when Internal is checked), and additionally creates a linked `deals` row if Target is checked (skipped when Internal) and/or an `enablers` row if Enabler is checked.
- **Editing an institution** (`saveInstitution`) ensures the org row exists and reconciles the checkboxes: checking Target/Enabler creates the deal/enabler, unchecking archives it via `purgeDeal`/`purgeEnabler` (deletes the deal/enabler and its junctions, but preserves shared activities/todos by nulling their FK). The edit modal confirms before an uncheck removes a linked row.
- **Linking a person to an institution** uses `institutionPrimaryEntity(inst)`: a Target links via its deal (so the person also appears on the Pipeline deal sheet), an Enabler via its enabler, otherwise via its organization. `institutionPeople(inst, ...)` unions people across all three backing rows plus legacy junctions.
- Clicking an institution anywhere, **including a deal card in the Pipeline**, opens the one **unified Institution Sheet** (keyed by name via `institutionSheetKey`). There is no separate Deal Sheet component: a pipeline deal is just an institution backed by a deal row, so it gets the same sheet with all features (People, To-Dos, collapsible AI Summary, Notes, Activity Timeline, Quick Add, Paths In, Connected Institutions). `sheetOrigin` ("pipeline"/"network") only controls the Back button label. Clicking a person opens the **Person Sheet** (keyed by contact id via `personSheetId`).

## Feature map

- **Calendar** (`CalendarTab`, nav id `calendar`, sidebar below Home; on mobile reached via the "Open Calendar" action on Home, no bottom tab): the Google Calendar, read from `calendar_events` (populated hourly by the Apps Script; the app never creates events). Two modes toggled in the header: **Agenda** (default) lists upcoming events grouped by day (Today, Tomorrow, then future days) with the last 2 days collapsed under a "Recent" section; each `CalendarEventRow` shows time range, title, location, attendee names, a clickable matched-entity pill (blue deal / gold enabler / green contact / gray org via `eventMatchClass`), and a "Prep Brief" button whenever the event is matched to ANY entity (including an internal team contact) or `isInternalMeeting` detects every attendee is internal team (an @mangosciences.com email, or a contact flagged `is_internal`), showing an "Internal meeting" tag when that fallback applies with no FK match at all. **Week** is a Mon-Sun grid (`startOfWeek`, prev/next/today nav) with events as colored blocks by matched type; clicking a block opens a details popover with the same row + Prep Brief. Unmatched, non-internal events show a "Link to entity" `EntityPicker` (deduped institutions + contacts) that calls `linkCalendarEvent` to set the single matched_* FK. "Prep Brief" (`prepBriefForEvent`) reuses the existing meeting-brief generation with the event's matched contact/deal/enabler/org plus title and time, auto-picking External or Internal via `detectBriefType` (see Meeting Prep Briefs); reuses a same-title-same-day brief if one exists. A Refresh button re-reads via `refreshCalendar` and a "Last synced" time shows `max(updated_at)`. Hidden actions in Boss View (read-only). Home's Today's Agenda reads the same `calendar_events` with the same Prep Brief gating.
- **Home / Command Center** (`HomeTab`, nav id `home`, the default view and the mobile landing tab): a morning briefing. Header greeting (morning/afternoon/evening by clock, "Fahed" or "Andy" in Boss View) plus today's date. Sections, all derived in `App` and passed in: (1) **Unread Comments**, boss_comments from the other person where `is_read = false`, each with avatar/text/timestamp, a clickable entity tag, and a "Mark read" button (`markCommentRead` PATCHes `is_read`); (2) **Today's Agenda**, calendar_events whose start_time is today (time, title, location, matched entity, and a "Prep Brief" button when matched), plus an "Open Calendar" link, else "No events on your calendar today."; (3) **Daily Briefing** (see below); (4) **Briefs**, the 5 most recent meeting_briefs plus "+ New Brief"; (5) **Urgent Tasks**, top 5 open todos ranked overdue then due-today then high-priority (left border red/amber), each with a completion checkbox (hidden in Boss View) and clickable only when it links to an entity; (6) **Recent Activity**, last 10 activities with glyph, description, entity name, timestamp; (7) **Stale Deals**, active deals with no activity in 14+ days. `entityName(row)`/`openEntity(row)` resolve any deal_id/enabler_id/organization_id/contact_id to a name and a navigation target. On mobile, "View Materials" and "View Reports" buttons sit at the bottom (neither has a mobile tab).
- **Daily Briefing** (`DailyBriefing` in `App.jsx`, on Home below Today's Agenda): three collapsible news sections (Health Tech and AI, Oncology and Immunotherapy, Saudi Arabia), each fetched by `fetchNewsStories` in `anthropic.js` (claude-sonnet-4-6 + web_search tool, max_tokens 1500, 90s timeout, JSON-array-only response `[{headline, summary, source, url}]` parsed via `parseJsonArray`). Each story row is a clickable `<a target="_blank" rel="noopener noreferrer">` that opens the source article. Article URLs are resolved deliberately (the model tends to return a publication homepage instead of the story): the prompt demands the DIRECT article link (or null); `fetchNewsStories` uses `webSearchTextWithSources` to also harvest the real result URLs from the `web_search_tool_result` blocks (and text-block citations), and `pickArticleUrl` keeps the model's url only when it is a deep link, otherwise matches the story to a result URL by headline/title word overlap plus a source-domain bonus. `isHomepageUrl` flags a bare-domain URL (no path); anything still resolving to a homepage falls back to a Google search link for the headline. `newsStoryHref` re-applies that homepage guard at render time (protecting older cached payloads). The three fetches are **staggered by 1s** (fire, +1s, +2s) so simultaneous web searches do not rate-limit each other, and each is **resilient**: `fetchSection` retries up to 2 times with a 2s backoff, then falls back to `fetchNewsStoriesNoSearch` (a plain, no-web-search call on the model's own knowledge using the section's `fallbackPrompt`) before surfacing an error. Every attempt's real error is `console.warn`/`console.error`-logged (`[News] <section> attempt n/3 failed: ...`). `parseJsonArray` also does **partial recovery**: if a response is truncated by max_tokens (no closing `]`), `recoverJsonObjects` salvages every complete `{...}` object rather than failing. A failed section shows "Couldn't load [section] news. Tap to retry." (the button shows "Retrying..." and re-attempts just that one section) without touching the others; loading shows shimmer skeletons. Results cache in localStorage (`mango-news-briefing-v3`, keyed by date) so news is fetched once per day (first Home mount of the day) or on the explicit "Refresh Briefing" button; a "Last updated" timestamp shows next to it. (The cache key is bumped whenever the story shape changes so stale payloads are discarded: `-v2` added URLs, `-v3` corrected them to deep links; the older `mango-news-briefing` and `-v2` keys are removed once on mount.) Sections are expanded by default on desktop, collapsed on mobile. **Boss View never fetches**: Andy's browser must not spend web-search API calls on Fahed's key, so it shows any cache that exists else a static "Daily news is fetched on Fahed's device." note, with the Refresh button hidden. Prompt config lives in `NEWS_SECTIONS` in `App.jsx`.
- **Materials Library** (`MaterialsTab`, nav id `materials`): a card grid of sales collateral. Each card shows name, type badge, audience tag, version, file size, upload date, and Download / Edit (inline name/type/audience/version) / Delete actions. "+ Upload Material" expands a form: file picker (PDF/PPTX/DOCX/images), name auto-filled from the filename (editable), type, audience, version, notes. Files are converted to a base64 data URL client-side and capped at 5MB ("File too large. Keep materials under 5MB."). Institution sheets get a **Materials section**: "Attach Material" opens a picker over the library (attach creates a material_links row via the institution's primary entity), each attached row shows name, version, a download icon, and an X to unlink. Deleting a material cascades its material_links first.
- **Meeting Prep Briefs**: "Prep Brief" on a Today's Agenda meeting or Calendar event (reuses an existing brief with the same title on the same day instead of regenerating) or "+ New Brief" on Home (modal: contact and/or institution, title, date). There are two brief types, stored in `meeting_briefs.brief_type` (`external` default, or `internal`), auto-selected by `detectBriefType(ev)` when generating from a calendar event: a match to an external deal/enabler/organization, or to a contact who is not internal, always wins as **External**; otherwise, if every attendee on the event is internal team (`isInternalMeeting`), it is **Internal**. The user can override the guess from the brief itself. **External** (`generateMeetingBrief` in `anthropic.js`, max_tokens 600): `gatherBriefContext` assembles contact info/roles/warmth, last 10 activities with the contact and on the deal/institution, open tasks, ai_summary, linked materials; renders as WHO / CONTEXT / OPEN ITEMS / SUGGESTED TALKING POINTS. **Internal** (`generateInternalBrief`, max_tokens 600, "this is a sync with my own team, not an external prospect"): `gatherInternalBriefContext` is a status roll-up, not tied to one contact, pulled fresh from already-loaded state: deals with `last_activity_at` in the last 7 days (movement), deals `created_at` in the last 7 days (new), deals with no activity in 14+ days that aren't won/lost (stale), the last 20 activities on any deal or enabler in the last 7 days, open high-priority tasks, overdue tasks, unresolved (`!is_read`) `boss_comments` from Andy, and deals in Negotiation or Proposal stage; renders as PIPELINE MOVEMENT / DECISIONS NEEDED / BLOCKERS / HIGHLIGHTS. `BriefModal` shows a small type badge ("External Meeting Brief" blue / "Internal Team Brief" purple) plus a "Switch to External/Internal" link that regenerates the same brief as the other type (`switchBriefType`), formatted by `FormattedActivityBody` either way (both prompts contract to exactly four heading lines), an editable "My Goal for This Meeting" (PATCHes my_goal on blur), "Related Materials" (materials linked to the brief's deal/institution, with download), and Delete / Regenerate (Regenerate keeps the brief's current type). A print stylesheet makes the open brief printable. Generation is hidden in Boss View (briefs remain viewable).
- **Pipeline**: Kanban board of deals by stage, drag-and-drop between columns, "+ New Deal". The default "All" view groups deals **by tier** within each stage column ("Tier 1" / "Tier 2" / "Tier 3" / "Untiered" section headers); the tier filter buttons at the top collapse the board to a single tier. Each deal card shows the company name, a tier badge (exact tokens: T1 `#FDECCB`/`#B77400`, T2 `#E4EDFB`/`#2A6FDB`, T3 `#ECECEF`/`#6B6B7B`), a color-coded institution type badge (from the linked org), an inline-editable city (pin, "first + N more" for multi-city), and value. Clicking a card opens the unified sheet (`renderDealCard` in App).
- **Ecosystem tab** (`NetworkTab`, nav id `network`, label "Ecosystem"): the master ecosystem view. Top bar = search + "+ Institution" + "+ Person" (each expands an inline form with the Target/Enabler checkboxes). Two sub-tabs:
  - *Institutions*: type/city filters plus "Targets only"/"Enablers only" toggles; a card grid grouped by type. Each card shows name, an inline-editable type badge (`InlineSelectField`, includes custom types + "+ Add custom"), Target (blue) / Enabler (gold) badges, an inline-editable city (`InlineCity`), pipeline stage (if Target), people count, and a preview of the first few people with roles.
  - *People*: a **Cards / Table** view toggle (default **Table**). The table (`PeopleTable`) has columns Name, Role, Institution, City, Email, Phone, LinkedIn, Warmth, Outreach, Last Contacted; every cell is inline-editable, Enter/Tab commits and moves focus **down the same column** (rapid data entry), columns are sortable (default institution then name). Filters/search apply to both views.
- **Unified Institution Sheet** (`InstitutionSheet`): the single sheet for every institution (Targets/deals, Enablers, plain orgs). Header: inline-editable name, inline-editable type badge, Target (blue) / Enabler (gold) badges, an inline tier selector (if Target), a multi-city `CityEditor` (removable pins + "Add city"), Target/Enabler checkboxes (checking creates the deal/enabler carrying city over, unchecking archives it), inline sector/website, "Auto-fill with AI" and Delete. If Target, the stage row also has inline **Value (USD)** and **Next action** fields (saved to the deal), and the header has an inline Region field. Sections: **collapsible AI Summary** (`SummaryCard`, default collapsed to a 2-line preview with a chevron and Regenerate; summaries are generated on demand only, via the empty-state "Generate summary" button or Regenerate, never automatically on sheet open, so opening a fresh sheet costs no API call), Description, always-on Notes, "People at <name>" (+ "Research Key People" web search), To-Dos, Connected Institutions, "Paths In" (Targets only), Quick Add, Activity Timeline. Every field is click-to-edit (no edit modal).
- **Person Sheet**: header with an inline-editable name, warmth `BadgeSelect`, an **Internal Team checkbox** (auto-tags "Internal Team"; drives the Network Map's Internal nodes), inline email/phone/linkedin/**source**, and an editable **tags** row (toggle opens `TagPickerWithCustom`); collapsible AI Summary; a "Profile" section parsed from researched notes; always-on Notes; Roles; Connections; Quick Add; Activity Timeline. All fields are inline-editable; the old edit-contact modal was deleted.
- **Network Map tab** (`src/MapTab.jsx`, nav id `map`, label "Network Map"): a full-screen D3 force-simulation graph of the whole ecosystem, rendered as SVG (D3 owns the DOM inside a ref layer; React owns the top bar, side panel, tooltip, and paths panel). `buildGraph` turns the derived institutions and contacts into nodes and dedupes edges from contact_roles / deal_contacts / enabler_contacts / deal_enablers / network_edges. It reuses the already-loaded App data via props rather than re-fetching. Two modes: **Orbit Map** (radial layout, targets pulled to the center, enablers to a middle ring, others outer, people orbiting via link forces) and **Path Finder** (pick a Target; a depth-limited search from every known/internal person surfaces up to 4-hop chains, highlighted in gold with a left-to-right layout and a clickable paths list). Fresh nodes (activity in 7 days) pulse; stale nodes (30+ days) fade. Hover highlights neighbors and shows a tooltip; clicking a node opens a side panel (people, connections, recent activity, "Open Sheet" which routes to the Institution/Person sheet); nodes are draggable; the canvas zooms/pans with wheel and top-bar controls. Type filters (Targets/Enablers/Competitors/People/Internal) fade nodes to near-zero opacity rather than removing them, preserving the layout. The simulation caps at 300 ticks.
- **Tasks tab**: every open todo, sorted priority-then-due-date, filterable, inline-editable (click the edit pencil). A task can link to any combination of a deal, an enabler, an organization, and a contact. One shared `TaskForm` is used everywhere (Tasks-tab quick-add, sheet task sections, and inline edit): title, priority, due date, and three searchable `EntityPicker` link fields (Deal, Institution = enablers + organizations, Contact) that pre-fill from context via an `initial` prop and can be added to or unlinked. `TaskPills` renders every linked entity below the task title as a clickable pill that navigates to that sheet, color-coded blue (deal) / gold (enabler) / green (contact) / gray (organization). A "+ Task" button appears in the To-Dos section on institution sheets (pre-fills the institution's primary deal/enabler/organization) and in a new "Tasks" section on person sheets (pre-fills the contact, shows todos where contact_id matches with the same collapsible completed pattern). On a Person Sheet, `contact_id` is force-set on every save (survives even if the Contact picker is cleared) and the Institution picker pre-fills to the contact's primary role (`contact_roles.is_primary`, falling back to the first resolved role) so a task added from a contact card lands on both that person's and their institution's Tasks/To-Dos without any manual linking. `openTaskLink` handles all four entity types.
- **Notes tab** (`NotesTab`, nav id `notes`, between Tasks and Reports in the sidebar; hidden in Boss View since notes are Fahed's personal workspace): a two-panel note-taking surface. Left list panel: search over title+content, "+ New Note" and "+ New Folder" buttons, then a **folder tree**. Order is a "📌 Pinned" section (all pinned notes, regardless of folder), the folder tree, then an "Unfiled" section (notes with no folder). Folders are collapsible (chevron + folder icon; expanded state persists in localStorage under `mango-note-folders-expanded`, collapsed by default), nest via `parent_id` with indentation, are sorted by sort_order then name, and show a recursive non-pinned note count. A note item shows title, first non-empty content line, updated date, and a linked-entity tag. Folder management: "+ New Folder" creates one with an inline-editable name; each folder row has a three-dot menu (and desktop right-click) with Rename (inline), Create subfolder, and Delete (moves notes to Unfiled). The recursive tree is rendered by a plain `renderFolder` function (not a component) so inline rename inputs keep focus across re-renders. Notes are moved between folders by drag-and-drop onto a folder row or the Unfiled section (desktop; note items are `draggable`), or via a folder-picker dropdown in the editor toolbar. Right editor panel (`NoteEditor`): large inline title input (saves on blur, placeholder "Untitled"), a plain-text content textarea with 1.7 line-height that autosaves on a 1s debounce (and flushes on blur) with a subtle "Saved" indicator, then a toolbar with Pin/Unpin, a "Link to..." searchable dropdown over deals/enablers/organizations/contacts (single-link, shown as a removable pill via `linkNote`), a "📁 Folder" move dropdown (shows the flattened folder tree, calls `moveNoteToFolder`), and Delete (confirm). Both toolbar dropdowns open **upward** (`bottom: 100%`) since the toolbar sits at the bottom of the editor and a downward menu would be clipped off-screen. "+ New Note" (`createNote`) inserts an Untitled row and opens it with the title focused. On mobile the list and editor are separate full-screen views (`.notes-editing` toggles them; the editor gets a "← Notes" back button), and the tab is reached from a "My Notes" section on the Home screen (there is no Notes bottom tab). A read-only "Linked Notes" section (`LinkedNotesSection`) on Institution and Person sheets lists notes linked to that entity (by deal_id/enabler_id/organization_id/contact_id); clicking one calls `openNote` to jump to the Notes tab.
- **Outreach Engine** (`OutreachTab`, nav id `outreach`, hidden in Boss View): two sub-tabs. *Follow-ups* (default) groups contacts by outreach state: **NEEDS NUDGE** (red, awaiting_reply for `NUDGE_AFTER_DAYS` = 5+ days, "Draft Nudge" opens the compose panel preloaded with the Re-engagement template), **AWAITING REPLY** (yellow, under 5 days), **RECENTLY REPLIED** (green, status replied within the last 7 days: reply detection and manual switches stamp last_contacted_at so the group ages out), and **NOT YET CONTACTED** (gray, untouched contacts tied to a Tier 1 target via its deal, organization, or enabler row, or warmth warm/hot, with a "Compose" button). Every row shows institution, days waiting, last outreach date, and an inline status dropdown (`setOutreachStatus`; moving into awaiting_reply stamps the wait clock). *Templates* lists email_templates with category badges and an inline editor showing a merge-field legend. **Compose panel** (`ComposeModal`, App-level, also opened by the "✉ Compose Email" button next to the email on Person Sheets): pick a template, merge fields fill from the contact via `fillTemplate` ({my_note} is left in place with a yellow warning until replaced), then "Open in Gmail" (mail.google.com compose URL, disabled with a warning when the contact's email fails a basic format check) or "Copy to Clipboard". Either action calls `recordOutreach`: contact goes to awaiting_reply with last_outreach_at/awaiting_reply_since/last_contacted_at stamped, and an activity "Sent outreach: [subject]" is logged (that prefix is the contract that keeps reply detection from counting our own sends). **Reply detection**: an effect on load flips awaiting_reply contacts to replied when a type-email activity from them (not "Sent outreach:") is newer than awaiting_reply_since (ref-guarded, skipped in Boss View). The People table has an **Outreach** column (colored badge select), and Home shows an "X contacts need a nudge" banner in Urgent Tasks linking to Follow-ups.
- **Reports**: plaintext EOD/EOW report generation + copy-to-clipboard.
- **Boss View**: see Navigation above. It reuses every normal view read-only (no separate dashboard component); the only Boss-specific UI is the sidebar user card (Andy Liu / Read-only) and the per-entity "Boss Notes" section on the unified sheet.
- **Quick Add** (on every sheet): logs an activity. `ACT_TYPES` in `constants.js` is the canonical, stable lowercase id list: call, email, meeting, whatsapp, linkedin, note, proposal, demo, voice_note, transcript (the "Paste Transcript" UI mode). The dropdown always saves the id, never the label; an unknown or legacy id (e.g. old data) renders gracefully via a default dot glyph and its raw type text rather than breaking. WhatsApp and LinkedIn get their own `ActivityGlyph` colors, `#25D366` and `#0A66C2`. Supports a camera button (photo of handwritten notes to Claude vision to transcribed note) and a "Paste Transcript" type (textarea + "Summarize with AI"). The Activity Timeline's type filter (`TIMELINE_TABS`) includes WhatsApp and LinkedIn tabs alongside Calls/Emails/Meetings/Notes. **Cross-linking**: institution sheets get an optional "With person" picker (the institution's own people) and person sheets an "At institution" picker (the person's roles); the activity then carries both the institution FKs and contact_id so it appears on both timelines and bumps last_contacted_at. On a person sheet the institution picker defaults to the contact's primary role (falling back to their first resolved role) so it cross-links automatically without the picker being touched; picking a different role's institution overrides that default, and `addActivity` re-derives/stores the FK server-side as a safety net regardless of what the UI sent (see activities above). **Entity pills**: every activity row (deal/enabler/organization/person timelines, Home's Recent Activity, and the EOD report's per-line label) shows both the linked person and linked institution as small clickable pills (`ActivityEntityPills`), person pills neutral and institution pills colored by kind (blue deal, gold enabler, gray organization); a timeline hides the pill for the entity it already belongs to (a person's own timeline drops their name pill, an institution's own timeline drops its own pill). For older rows that never got an institution FK stored, the institution pill is derived at display time from the contact's current primary role rather than left blank.
- **Global search** (`GlobalSearch`, Cmd+K / Ctrl+K, the sidebar Search button, or the magnifier on Home): a client-side overlay over already-loaded data, grouped results (5 per group) across institutions, people, activities, tasks, notes, and materials; click navigates, Enter opens the top result, Esc closes.
- **Theme**: light only (the warm cream design system). Dark mode has been removed.
- **Warmth**: per-contact temperature (unknown/cold/warm/hot/active), a colored dot on people cards and person sheets.

## Conventions worth knowing before editing

- **Custom dropdowns**: every categorical dropdown uses `SelectWithCustom` (drop-in `<select>` that appends "+ Add custom..."; picking it swaps to a text input). To persist custom values, pass `options={optionsWithCustom(DEFAULTS, customOptions, "field_name")}` and wrap `onChange` with `trackCustom("field_name", thatSameOptionsList, onAddCustomOption)`. `customOptions` (raw rows) and `onAddCustomOption` (`App`'s `addCustomOption`) are threaded down as props. Field names in use: institution_type, city, region, deal_stage, warmth, relationship, activity_type, priority, tag, strength, enabler_type. Multi-select tags use `TagPickerWithCustom`; colored button groups (warmth) use `ButtonGroupWithCustom`. All live near the top of `App.jsx`.
- Every "delete" CRUD function manually cascades related rows first, since there is no guaranteed `ON DELETE CASCADE`. The cascade covers ALL dependent tables: junctions (deal_contacts/enabler_contacts/contact_roles/deal_enablers), network_edges (both directions), material_links (deleted), and history rows (activities, todos, notes, meeting_briefs, boss_comments, calendar_events' matched_* links) which are PATCHed to null the dangling reference so their content survives. `deleteContact` also strips the denormalized contact_id/contact_name/contact_role copies on deals and enablers. `deleteFullyOrphanedActivities` sweeps activities whose four entity FKs are all null. When adding a new table with entity FKs, extend deleteContact, purgeDeal, purgeEnabler, and deleteInstitution.
- **Targeted state updates**: high-frequency writes (inline field saves, stage/tier moves, todo add/toggle/update, activity logging, outreach recording, comments, custom options, AI summaries) PATCH/POST and then patch local React state; they must NOT call `loadData()`. Structural operations (create/delete entities, role changes, research) still do a full reload. A task checkbox toggle is exactly 1 HTTP request. The "Refresh data" button (sidebar) / refresh icon (Home) is the manual way to pull external changes (Gmail sync, Boss View edits).
- `persistContact`'s company auto-link dual-writes contact_roles (source of truth) plus the legacy junction row; keep any new people-linking write path doing the same (see `persistPersonRole`).
- Colors for enum-like fields are defined once in `constants.js` as `{ id, label, color }` and rendered as `<span className="badge" style={{background: color+"22", color, border: \`1px solid ${color}44\`}}>`. Reuse this rather than inventing badge styles.
- Location fields use `SelectWithCustom` against `CITY_OPTIONS`/`REGION_OPTIONS`. Render a city as a small pin, `📍 {city}`, with the `.city-pin` class.
- Only send optional string fields to Supabase when they have real content (trim and skip empties); never send an empty string for a numeric column.
- Never use the em dash character anywhere: not in UI text, labels, toasts, descriptions, generated content, AI prompts, or code comments. Use commas, periods, colons, or parentheses instead. The AI prompts explicitly instruct the model to avoid them too.
- No test suite exists. Verify changes by running the dev server and exercising the feature directly. This app talks to a live, shared Supabase project with real data, so clean up any test data afterward, and do not assume a row is test data just because it appeared recently; verify before deleting.

## Mobile responsiveness

The app is responsive at `max-width: 768px` (phones); most of the mobile layout lives in one media block at the bottom of `src/styles.css`. Key behaviors: the left sidebar is hidden and replaced by a fixed bottom **tab bar** (`MobileTabBar` in `App.jsx`, rendered next to `CommentPanel`; **Home/Pipeline/Ecosystem/Tasks/Outreach**, Outreach hidden in Boss View, active tab in mango gold, `env(safe-area-inset-bottom)` for notches). Global search is reached from the magnifier on Home. The Network Map is **desktop-only**: it is dropped from the mobile tab bar, and if `view === "map"` on a phone the map renders a "Network Map is best viewed on desktop." notice (`.map-mobile-notice`) instead of the D3 graph. Reports, Notes, and Materials have no mobile tab either; Reports and Materials are reached from "View Reports" / "View Materials" buttons at the bottom of Home, and Notes from a "My Notes" section on Home (the Notes list and editor are separate full-screen views on mobile, the editor gaining a back button). The Daily Briefing news sections start collapsed on mobile. The floating comment button sits just above the tab bar, right-aligned (`right: 16px; bottom: calc(56px + env(safe-area-inset-bottom) + 12px)`), with the Ecosystem add FAB stacked above it; `.main` gets extra bottom padding so content scrolls clear of both. These two floating-button overrides live in a final media block at the very end of `styles.css` because the base `.comment-fab`/`.comment-panel` rules are defined after the main responsive block and would otherwise win the cascade. The Pipeline shows one full-width stage column at a time with CSS scroll-snap; `MobilePipelineNav` (a header with prev/next arrows over a `kanbanRef`) tracks and scrolls the centered column. The Ecosystem "+ Institution"/"+ Person" buttons collapse into a floating add button (`.mobile-add-fab` in `NetworkTab`); People forces card view (`useIsMobile()` -> `effectivePeopleView`) with a "Switch to desktop for table editing" note. Sheets/modals go full width (modals become full-screen sheets, inputs 16px to avoid iOS zoom), stats become a 2x2 grid, tap targets are >= 44px, and the Network Map side panel becomes a 60% bottom sheet with larger person node tap targets (min 16px radius) and `touch-action: none` for pinch/pan. The floating comment button moves above the tab bar (bottom-left, clear of the add FAB). `useIsMobile(768)` is a small `matchMedia` hook for the few JS-driven switches CSS cannot do.

## Voice notes (Quick Add)

Every `QuickAdd` (person and institution sheets, desktop and mobile) has a microphone button beside the activity-type selector. It records via the Web Speech API (`window.SpeechRecognition || webkitSpeechRecognition`, `lang: "en-US"`, `continuous: true`, `interimResults: false`); the raw transcription is accumulated in a ref and never shown. On stop, the raw text goes to `digestVoiceNote(transcript)` in `src/anthropic.js`, which returns `{ summary, action_items: [{title, priority, due_date_hint}] }` (JSON only, max_tokens 500, backticks stripped before parsing). The result renders as a "Meeting Note" card: an editable summary textarea plus a checklist of action items with priority badges. "Save Note" logs an activity of type `voice_note` (a monochrome glyph in `ACTIVITY_GLYPHS`, matching the other timeline glyphs); "Save Note + Create Tasks" also creates todos linked to the sheet's entity via `onCreateTasks` (institution sheet uses its `onAddTodo`; person sheet gets an `onAddTodo` that stamps `contact_id`). `parseDueHint()` best-effort converts a due-date hint ("tomorrow", "next Thursday", "end of week") to an ISO date, otherwise the hint stays in the task title. If the browser lacks speech support, it shows "Voice notes not supported in this browser. Try Chrome or Safari."

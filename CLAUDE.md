# Mango OS

A single-page BD/pipeline command center built around one idea: everything the user tracks (a deal, an enabler, a competitor, a regulator, a hospital) is an institution in the Saudi healthcare ecosystem, and people hold roles at those institutions. Built as a personal tool, not a multi-tenant product: no auth, no row-level access control.

## Stack

- React 19 + Vite, plain CSS (`src/styles.css`) with CSS custom properties for theming. No component library, no CSS-in-JS.
- Supabase (Postgres + PostgREST) as the only backend. `src/supabase.js` exports a single `api(table, method, body, query)` helper that hits the REST API directly with the anon key; there is no server code in this repo.
- Anthropic API called directly from the browser (`src/anthropic.js`) for AI summaries, photo-note transcription, and transcript summarization. Requires `VITE_ANTHROPIC_API_KEY` at build time (see `.env.example`). There is no backend proxy, so the key is present in the built JS bundle. Acceptable for this project's threat model; revisit if that changes.
- `src/gmail-sync-updated.js` is a **reference Google Apps Script**, not part of the Vite app: it's a separate script pasted into Google Apps Script directly, used to auto-log Gmail activity into the `activities` table on a timer.

Almost all UI logic lives in one file, `src/App.jsx`: the top-level `App` component holds all state and CRUD functions, and every view/sheet/modal is a function component defined below it in the same file. `src/constants.js` holds the enum-like arrays (stages, types, priorities, cities, etc.) that drive dropdowns and color-coded badges throughout.

## Database schema (Supabase)

Core entities:
- **deals**: id, company, stage, contact_id, contact_name, contact_role, value, notes, next_action, city, region, created_at, last_activity_at, ai_summary, ai_summary_updated_at
- **contacts**: id, name, role, company, email, phone, linkedin, source, notes, tags[], warmth, is_internal, last_contacted_at, created_at, ai_summary, ai_summary_updated_at
- **enablers**: id, name, type, contact_id, contact_name, notes, city, region, created_at, last_activity_at, ai_summary, ai_summary_updated_at
- **organizations**: id, name, type, sector, description, website, notes, city, region, country (default "Saudi Arabia"), address, created_at, ai_summary, ai_summary_updated_at
- **activities**: id, deal_id, contact_id, enabler_id, organization_id, type, description, created_at. The shared activity-timeline log; any subset of the four FKs can be set.
- **todos**: id, title, description, priority, status, due_date, deal_id, enabler_id, contact_id, created_at, completed_at. No `organization_id` column; the Organization Sheet intentionally has no To-Dos section for this reason.

Junction / relationship tables:
- **contact_roles**: id, contact_id, entity_type ("deal"/"enabler"/"organization"), entity_id, role_title, is_primary, created_at. The current source of truth for "who belongs to which institution and in what capacity." Written and read by the Network tab's "+ Person" form and every sheet's "Add Person"/"Key People" section, via `addPersonRole`/`persistPersonRole`/`removePersonRole` in `App.jsx`.
- **deal_contacts** / **enabler_contacts**: deal_id/enabler_id, contact_id, role_in_deal/role_in_org (unique per pair). Predate `contact_roles`; every write to `contact_roles` for entity_type "deal"/"enabler" is fanned out to these tables too (and every write from these tables' own add-person flows is fanned out to `contact_roles`), so the older Deal/Enabler Sheet People sections keep working unchanged. Removing a role cleans up both sides.
- **deal_enablers**: deal_id, enabler_id, relationship (can_introduce/active/institutional), strength (strong/medium/weak/unknown), notes. Which enablers can help with which deals.
- **network_edges**: source_type, source_id, target_type, target_id, relationship, strength, direction, notes. Generic polymorphic graph edges for institution-to-institution connections (org-to-org, org-to-deal, org-to-enabler) and, for backward compatibility, contact-to-organization roles (every `contact_roles` write/delete for entity_type "organization" is mirrored here as a source_type="contact"/target_type="organization" edge, since the Organization Sheet's Key People section still reads this table). Relationship vocabulary: works_at, board_member, advisor, knows, can_introduce, reports_to, subsidiary, parent, board_overlap, partnership, competitor_to, regulates, funds, invested_in, custom.
- **custom_options**: id, field_name, value, created_at. Every dropdown built on `SelectWithCustom` (see below) persists a typed "+ Add custom" value here, keyed by a `field_name` string (e.g. "institution_type", "city", "region", "deal_stage", "enabler_type", "warmth", "priority", "tag", "activity_type", "relationship_type", "strength"), so it reappears as a normal option everywhere that field is edited from then on.

`deals`/`enablers`/`contacts`/`organizations` all carry `ai_summary`/`ai_summary_updated_at` and auto-regenerate their summary (via `SummaryCard`) whenever a sheet is opened and the existing summary is missing or older than the most recent activity. Summaries are also click-to-edit and independently saveable. All AI-generated prompts explicitly instruct the model not to use em dashes (see the no-em-dash rule below).

## The unified institution concept

The Network tab treats every deal, enabler, and organization as an "institution" with a `type` (see `INSTITUTION_TYPES` in `constants.js`: target, enabler, competitor, payer, government, regulator, association, research, hospital, plus whatever custom values have been added under `field_name: "institution_type"`). `institutionTypeMeta`/`normalizeTypeKey` in `App.jsx` resolve a raw type string to a canonical `{id, label, color}` case-insensitively, so a legacy free-typed value like "Hospital" still groups with the canonical "hospital" entry rather than creating a visually duplicate group.

Creating an institution (`addInstitution`) routes by type: "target" creates a row in `deals` (stage "prospecting"), "enabler" creates a row in `enablers`, anything else creates a row in `organizations`. This keeps the Pipeline tab (deals, unchanged) and Enablers tab (enablers, unchanged) in sync with whatever gets added from the Network tab. There is deliberately no separate "Institution Sheet" component: clicking an institution card routes to the existing `DealSheet`, `EnablerSheet`, or `OrganizationSheet` depending on its kind, since those already have the People/Connected/Paths In/Quick Add/Timeline/Summary structure the Network tab's detail view needs.

Adding a person (`addPersonWithRoles`, from the Network tab's "+ Person" form) creates a `contacts` row plus one `contact_roles` row per institution picked (a primary role plus any "+ Add another role" rows). Every place that resolves "who works where" (institution card previews, Key People sections, `ContactSheet`'s Institutions section, `findNetworkPaths`) reads `contact_roles` first and falls back to the older junction tables/`network_edges`/legacy `contact_id` fields for rows written before `contact_roles` existed.

`OrganizationSheet` additionally auto-detects "Also Connected (via shared people)": if a contact holds roles at this organization and at least one other institution, that other institution is surfaced as an implicit connection ("Connected to NUPCO through Khalid Al Rajhi") even with no explicit `network_edges` row between them. `DealSheet`'s "Paths In" (`findNetworkPaths`) does the same 1-2 hop traversal over both `network_edges` and `contact_roles`-implied edges, rendered as a "You > A (via) > B (via) > Deal" chain.

## Feature map

- **Pipeline**: Kanban board of deals by stage, drag-and-drop between columns. Deal cards show a city pin when `city` is set.
- **Deal Sheet / Enabler Sheet / Contact Sheet / Organization Sheet**: full-page detail views (not modals) sharing the same structural pattern: header info + Edit/Delete/Back, AI Summary card, People/Linked-entity sections, To-Dos (except Organization Sheet), Quick Add, and an Activity Timeline. The timeline component (filters: All/Calls/Emails/Meetings/Notes) is intentionally identical across all sheets.
  - Deal Sheet additionally shows "Paths In" and "Enabler Paths" (read-only, from `deal_enablers`).
  - Enabler Sheet additionally shows "Connected Targets" (editable, from `deal_enablers`, add/remove via `AddConnectionModal`).
  - Organization Sheet additionally shows "Key People", "Also Connected (via shared people)", "Connected Organizations", "Connected Deals", and "Connected Enablers" (add/remove via `AddOrgLinkModal`).
  - Contact Sheet shows a single "Institutions" section (from `contact_roles` plus legacy fallbacks) instead of separate "Linked Deals"/"Linked Enablers" lists, since a person can now hold roles at any institution kind.
- **Network tab** (`NetworkTab`): the unified ecosystem view.
  - Top bar: a search box plus "+ Institution" and "+ Person" buttons, each expanding an inline form below (not a modal). "+ Institution" (`InstitutionForm`) collects name/type/city/region/sector/description/website. "+ Person" (`PersonForm`) collects name/role/institution/email/phone/warmth plus repeatable "+ Add another role" rows, and searches across deals/enablers/organizations for the institution picker (`buildInstitutionOptions`).
  - Below that, a type filter and city filter, then the directory itself: every institution grouped into sections by type ("Targets", "Enablers", "Payers", "Hospitals", etc., each header showing a pluralized label and count), rendered as a responsive card grid. Each card shows name, type badge, city pin, people count, and a preview of the first few people with their roles.
  - Clicking a card routes to `DealSheet`/`EnablerSheet`/`OrganizationSheet` per its kind; clicking a person anywhere routes to `ContactSheet`.
- **Tasks tab**: every open todo across deals/enablers, sorted priority-then-due-date, filterable (All/High Priority/Due Today/Overdue), inline-editable (title/priority/due date/linked deal-enabler).
- **Reports**: plaintext EOD/EOW report generation + copy-to-clipboard.
- **Boss View**: pipeline overview, Key Summaries (AI summaries for active deals/enablers), Action Items (top 10 priority todos).
- **Quick Add** (on every sheet): logs an activity; supports a camera button (photo of handwritten notes → Claude vision → transcribed `note` activity) and a "Paste Transcript" activity type (textarea + "Summarize with AI" that replaces the pasted text in place before saving).
- **Theme**: dark (default) / light toggle via the header gear icon, persisted to `localStorage`. Implemented with CSS variables on `:root`, overridden by `.dark-mode`/`.light-mode` classes applied to **both** `.app` and `document.body` (body needs its own class since it's an ancestor of `.app`, not a descendant, so it can't inherit `.app`'s variable overrides). There's also an `@media (prefers-color-scheme: light)` fallback on bare `:root` for contexts where no explicit class has been applied yet; the explicit body/app classes always take precedence over it via specificity.
- **Warmth**: per-contact relationship-temperature indicator (unknown/cold/warm/hot/active), shown as a colored dot on contact cards and person cards, and set via a colored-button selector in the contact/person forms.
- **Last synced indicator**: header text reading the most recent activity of type `email` or `meeting` and showing its timestamp, as a signal that the Gmail/Calendar sync is working.

## Conventions worth knowing before editing

- Every "delete" CRUD function manually cascades related rows first (activities, junction tables, `contact_roles`, `network_edges`, todos) before deleting the parent row, since there's no guarantee of `ON DELETE CASCADE` at the DB level. Follow the existing pattern in `deleteDeal`/`deleteEnabler`/`deleteContact`/`deleteOrganization` when adding new relationships.
- Shared UI pieces (`QuickAdd`, `TodoSection`/`TodoRow`, `PeopleSection`, `SummaryCard`, `AddConnectionModal`, `AddOrgLinkModal`) are deliberately factored out and reused across sheets rather than duplicated. Extend those instead of writing sheet-specific copies.
- Colors for enum-like fields (stage, enabler type, org/institution type, priority, strength, warmth) are defined once in `constants.js` as `{ id, label, color }` arrays and rendered as `<span className="badge" style={{background: color+"22", color, border: \`1px solid ${color}44\`}}>`. Reuse this pattern for any new categorical field rather than inventing new badge styles.
- Every dropdown backed by one of these enum-like arrays should use `SelectWithCustom` (a drop-in `<select>` replacement that appends a "+ Add custom..." option; picking it swaps the select for a text input committed on blur/Enter) instead of a plain `<select>`, so users can always type a value outside the predefined list. For multi-select tag pickers use `TagPickerWithCustom`, and for colored button-group pickers (like warmth) use `ButtonGroupWithCustom`. All three live near the top of `src/App.jsx`, above `export default function App()`. To make a dropdown's custom values persist, pass it `options={optionsWithCustom(DEFAULTS, customOptions, "field_name")}` and wrap its `onChange` with `trackCustom("field_name", thatSameOptionsList, onAddCustomOption)`; `customOptions` (the raw `custom_options` rows) and `onAddCustomOption` (`App`'s `addCustomOption` function) are threaded down as props from `App` through each sheet/form.
- Location fields (`city`, `region` on deals/enablers/organizations) use `SelectWithCustom` against `CITY_OPTIONS`/`REGION_OPTIONS` (derived from `SAUDI_CITIES`/`REGIONS` in `constants.js`, merged with any custom values). Render a city as a small pin, `📍 {city}`, with the `.city-pin` CSS class, on any card or row where it's set.
- Never use the em dash character anywhere: not in UI text, labels, descriptions, generated content, AI prompts, or code comments. Use commas, periods, colons, or parentheses instead. This applies to text written by hand and to instructions embedded in Anthropic API prompts (the summary/transcript/photo-note prompts explicitly tell the model not to use them).
- No test suite exists. Verify changes by running the dev server and exercising the feature directly. This app talks to a live, shared Supabase project with real data, so be careful not to leave test data behind when doing so, and don't assume a row is test data just because it appeared recently; verify before deleting.

# Mango OS

A single-page BD/pipeline command center for the Saudi healthcare ecosystem. The mental model: every hospital, VC, government body, payer, etc. is an **institution**, and **people** hold roles at institutions. Some institutions are also pipeline **Targets** (sales/BD deals) and/or **Enablers** (partners who can open doors). Built as a personal tool, not a multi-tenant product: no auth, no row-level access control.

## Stack

- React 19 + Vite, plain CSS (`src/styles.css`) with CSS custom properties for theming. No component library, no CSS-in-JS.
- Supabase (Postgres + PostgREST) as the only backend. `src/supabase.js` exports a single `api(table, method, body, query)` helper that hits the REST API directly with the anon key; there is no server code in this repo.
- Anthropic API called directly from the browser (`src/anthropic.js`) for AI summaries, photo-note transcription, and transcript summarization. Requires `VITE_ANTHROPIC_API_KEY` at build time (see `.env.example`). There is no backend proxy, so the key is present in the built JS bundle. Acceptable for this project's threat model; revisit if that changes.
- `src/gmail-sync-updated.js` is a **reference Google Apps Script**, not part of the Vite app: it's pasted into Google Apps Script directly to auto-log Gmail activity into the `activities` table on a timer.

Almost all UI logic lives in one file, `src/App.jsx`: the top-level `App` component holds all state and CRUD functions, and every view/sheet/modal is a function component defined below it. `src/constants.js` holds the enum-like arrays (institution types, stages, priorities, cities, etc.) that drive dropdowns and color-coded badges.

## Database schema (Supabase)

The **city** column on deals, enablers, and organizations stores either a plain
string (one city) or a JSON array string like `["Riyadh","Jeddah"]` (many). Use
`parseCities(city)` to read (always returns an array) and `serializeCities(list)`
to write (returns a plain string for one city, a JSON array string for several,
or null for none). Editing an institution's city syncs all its backing rows.

Core entities:
- **deals**: id, company, stage, **tier** ("Tier 1"/"Tier 2"/"Tier 3"/"Untiered", default "Untiered"), contact_id, contact_name, contact_role, value, notes, next_action, city, region, created_at, last_activity_at, ai_summary, ai_summary_updated_at
- **contacts**: id, name, role, company, email, phone, linkedin, source, notes, tags[], warmth, is_internal, **city**, last_contacted_at, created_at, ai_summary, ai_summary_updated_at
- **enablers**: id, name, type, contact_id, contact_name, notes, city, region, created_at, last_activity_at, ai_summary, ai_summary_updated_at
- **organizations**: id, name, type, sector, description, website, notes, city, region, country (default "Saudi Arabia"), address, **research_data** (JSON string from the "Research Key People" web search), created_at, ai_summary, ai_summary_updated_at
- **activities**: id, deal_id, contact_id, enabler_id, organization_id, type, description, created_at. Shared activity-timeline log; any subset of the four FKs can be set (an institution activity is logged with all of its backing FKs so it appears everywhere at once).
- **todos**: id, title, description, priority, status, due_date, deal_id, enabler_id, contact_id, created_at, completed_at.
- **notes**: id, title (default "Untitled"), content, tags[], **deal_id/enabler_id/organization_id/contact_id** (single-link: at most one is set), is_pinned, created_at, updated_at. Backs the Notes tab. There is no updated_at trigger, so every write stamps updated_at manually. Note writes patch local `notes` state instead of a full reload, so the editor stays snappy (no cursor jumps while typing).
- **boss_comments**: id, author, content, file_name, file_data (base64 data URL), **deal_id/enabler_id/organization_id** (optional entity tag), **is_read** (boolean, default false), created_at. Backs the two-way comment thread between Fahed and Andy: a floating comment button (`CommentPanel`, bottom right, on every tab in both modes) opens a slide-up panel; untagged rows are the general thread, and rows tagged with an entity id also appear in that institution's "Boss Notes" section (`BossNotes`). Fahed's floating button shows an unread badge (comments from Andy newer than a localStorage "last seen" timestamp). Separately, the `is_read` column is the persisted per-comment flag that drives the Home / Command Center "Unread Comments" list and its per-comment "Mark read" button (distinct from the floating badge's timestamp).

Junction / relationship tables:
- **contact_roles**: id, contact_id, entity_type ("deal"/"enabler"/"organization"), entity_id, role_title, is_primary, created_at. The source of truth for who belongs to which institution and in what capacity.
- **deal_contacts** / **enabler_contacts**: legacy per-institution people tables. Every contact_roles write for entity_type "deal"/"enabler" is dual-written here (and vice versa) so the Pipeline deal sheet keeps working. Removing a role cleans up both sides.
- **deal_enablers**: deal_id, enabler_id, relationship, strength, notes (legacy; read-only "Enabler Paths" on the deal sheet).
- **network_edges**: source_type, source_id, target_type, target_id, relationship, strength, direction, notes. Institution-to-institution edges (org-to-org, etc.), person-to-person connections, and (for backward compat) person-to-organization role mirrors. Relationship vocab in `NETWORK_EDGE_RELATIONSHIPS`.
- **custom_options**: id, field_name, value, created_at. Persisted "+ Add custom" dropdown values, keyed by field_name (see custom-dropdown convention below).

## The institution model (key concept)

An **institution** is a derived object, one per normalized (lowercased/trimmed) name, folding together up to three backing rows: an `organizations` row (its type/city/sector/etc.), a `deals` row (making it a **Target**), and an `enablers` row (making it an **Enabler**). `buildInstitutions(deals, enablers, organizations)` in `App.jsx` produces this list; each institution carries `orgId`/`dealId`/`enablerId`, `isTarget`/`isEnabler`, resolved `type`/`city`/`stage`, etc. Institution "type" (Hospital, VC, Government, Tech Company, Payer, Regulator, Association, Research, Pharmaceutical, plus custom) is separate from the Target/Enabler flags. `institutionTypeMeta`/`normalizeTypeKey` resolve a raw type string to a canonical `{id,label,color}` case-insensitively so legacy free-typed values group correctly.

- **Creating an institution** (`addInstitution`) always creates an `organizations` row, and additionally creates a linked `deals` row if Target is checked and/or an `enablers` row if Enabler is checked.
- **Editing an institution** (`saveInstitution`) ensures the org row exists and reconciles the checkboxes: checking Target/Enabler creates the deal/enabler, unchecking archives it via `purgeDeal`/`purgeEnabler` (deletes the deal/enabler and its junctions, but preserves shared activities/todos by nulling their FK). The edit modal confirms before an uncheck removes a linked row.
- **Linking a person to an institution** uses `institutionPrimaryEntity(inst)`: a Target links via its deal (so the person also appears on the Pipeline deal sheet), an Enabler via its enabler, otherwise via its organization. `institutionPeople(inst, ...)` unions people across all three backing rows plus legacy junctions.
- Clicking an institution anywhere, **including a deal card in the Pipeline**, opens the one **unified Institution Sheet** (keyed by name via `institutionSheetKey`). There is no separate Deal Sheet component: a pipeline deal is just an institution backed by a deal row, so it gets the same sheet with all features (People, To-Dos, collapsible AI Summary, Notes, Activity Timeline, Quick Add, Paths In, Connected Institutions). `sheetOrigin` ("pipeline"/"network") only controls the Back button label. Clicking a person opens the **Person Sheet** (keyed by contact id via `personSheetId`).

## Feature map

- **Home / Command Center** (`HomeTab`, nav id `home`, the default view and the mobile landing tab): a morning briefing. Header greeting (morning/afternoon/evening by clock, "Fahed" or "Andy" in Boss View) plus today's date. Five sections, all derived in `App` and passed in: (1) **Unread Comments**, boss_comments from the other person where `is_read = false`, each with avatar/text/timestamp, a clickable entity tag, and a "Mark read" button (`markCommentRead` PATCHes `is_read`); (2) **Today's Agenda**, meeting-type activities logged today, else "No meetings synced today. Check Google Calendar sync." (no calendar integration yet); (3) **Urgent Tasks**, top 5 open todos ranked overdue then due-today then high-priority (left border red/amber); (4) **Recent Activity**, last 10 activities with glyph, description, entity name, timestamp; (5) **Stale Deals**, active deals with no activity in 14+ days. `entityName(row)`/`openEntity(row)` resolve any deal_id/enabler_id/organization_id/contact_id to a name and a navigation target. On mobile a "View Reports" button sits at the bottom (Reports has no mobile tab).
- **Pipeline**: Kanban board of deals by stage, drag-and-drop between columns, "+ New Deal". The default "All" view groups deals **by tier** within each stage column ("Tier 1" / "Tier 2" / "Tier 3" / "Untiered" section headers); the tier filter buttons at the top collapse the board to a single tier. Each deal card shows the company name, a tier badge (exact tokens: T1 `#FDECCB`/`#B77400`, T2 `#E4EDFB`/`#2A6FDB`, T3 `#ECECEF`/`#6B6B7B`), a color-coded institution type badge (from the linked org), an inline-editable city (pin, "first + N more" for multi-city), and value. Clicking a card opens the unified sheet (`renderDealCard` in App).
- **Ecosystem tab** (`NetworkTab`, nav id `network`, label "Ecosystem"): the master ecosystem view. Top bar = search + "+ Institution" + "+ Person" (each expands an inline form with the Target/Enabler checkboxes). Two sub-tabs:
  - *Institutions*: type/city filters plus "Targets only"/"Enablers only" toggles; a card grid grouped by type. Each card shows name, an inline-editable type badge (`InlineSelectField`, includes custom types + "+ Add custom"), Target (blue) / Enabler (gold) badges, an inline-editable city (`InlineCity`), pipeline stage (if Target), people count, and a preview of the first few people with roles.
  - *People*: a **Cards / Table** view toggle (default **Table**). The table (`PeopleTable`) has columns Name, Role, Institution, City, Email, Phone, LinkedIn, Warmth, Last Contacted; every cell is inline-editable, Enter/Tab commits and moves focus **down the same column** (rapid data entry), columns are sortable (default institution then name). Filters/search apply to both views.
- **Unified Institution Sheet** (`InstitutionSheet`): the single sheet for every institution (Targets/deals, Enablers, plain orgs). Header: inline-editable name, inline-editable type badge, Target (blue) / Enabler (gold) badges, an inline tier selector (if Target), a multi-city `CityEditor` (removable pins + "Add city"), Target/Enabler checkboxes (checking creates the deal/enabler carrying city over, unchecking archives it), inline sector/website, "Auto-fill with AI" and Delete. If Target, a stage `BadgeSelect`. Sections: **collapsible AI Summary** (`SummaryCard`, default collapsed to a 2-line preview with a chevron and Regenerate), Description, always-on Notes, "People at <name>" (+ "Research Key People" web search), To-Dos, Connected Institutions, "Paths In" (Targets only), Quick Add, Activity Timeline. Every field is click-to-edit (no edit modal).
- **Person Sheet**: header with an inline-editable name, warmth `BadgeSelect`, inline email/phone/linkedin; collapsible AI Summary; a "Profile" section parsed from researched notes; always-on Notes; Roles; Connections; Quick Add; Activity Timeline. All fields are inline-editable (no edit modal).
- **Network Map tab** (`src/MapTab.jsx`, nav id `map`, label "Network Map"): a full-screen D3 force-simulation graph of the whole ecosystem, rendered as SVG (D3 owns the DOM inside a ref layer; React owns the top bar, side panel, tooltip, and paths panel). `buildGraph` turns the derived institutions and contacts into nodes and dedupes edges from contact_roles / deal_contacts / enabler_contacts / deal_enablers / network_edges. It reuses the already-loaded App data via props rather than re-fetching. Two modes: **Orbit Map** (radial layout, targets pulled to the center, enablers to a middle ring, others outer, people orbiting via link forces) and **Path Finder** (pick a Target; a depth-limited search from every known/internal person surfaces up to 4-hop chains, highlighted in gold with a left-to-right layout and a clickable paths list). Fresh nodes (activity in 7 days) pulse; stale nodes (30+ days) fade. Hover highlights neighbors and shows a tooltip; clicking a node opens a side panel (people, connections, recent activity, "Open Sheet" which routes to the Institution/Person sheet); nodes are draggable; the canvas zooms/pans with wheel and top-bar controls. Type filters (Targets/Enablers/Competitors/People/Internal) fade nodes to near-zero opacity rather than removing them, preserving the layout. The simulation caps at 300 ticks.
- **Tasks tab**: every open todo, sorted priority-then-due-date, filterable, inline-editable (click the edit pencil to edit title/priority/due date/linked deal-enabler).
- **Notes tab** (`NotesTab`, nav id `notes`, between Tasks and Reports in the sidebar; hidden in Boss View since notes are Fahed's personal workspace): a two-panel note-taking surface. Left list panel (search over title+content, "+ New Note", a "📌 Pinned" group then a "Recent" group sorted by updated_at desc; each item shows title, first non-empty content line as preview, updated date, and a linked-entity tag). Right editor panel (`NoteEditor`): large inline title input (saves on blur, placeholder "Untitled"), a plain-text content textarea with 1.7 line-height that autosaves on a 1s debounce (and flushes on blur) with a subtle "Saved" indicator, then a toolbar with Pin/Unpin, a "Link to..." searchable dropdown over deals/enablers/organizations/contacts (single-link, shown as a removable pill via `linkNote`), and Delete (confirm). "+ New Note" (`createNote`) inserts an Untitled row and opens it with the title focused. On mobile the list and editor are separate full-screen views (`.notes-editing` toggles them; the editor gets a "← Notes" back button), and the tab is reached from a "My Notes" section on the Home screen (there is no Notes bottom tab). A read-only "Linked Notes" section (`LinkedNotesSection`) on Institution and Person sheets lists notes linked to that entity (by deal_id/enabler_id/organization_id/contact_id); clicking one calls `openNote` to jump to the Notes tab.
- **Reports**: plaintext EOD/EOW report generation + copy-to-clipboard.
- **Boss View**: see Navigation above. It reuses every normal view read-only (no separate dashboard component); the only Boss-specific UI is the sidebar user card (Andy Liu / Read-only) and the per-entity "Boss Notes" section on the unified sheet.
- **Quick Add** (on every sheet): logs an activity; supports a camera button (photo of handwritten notes to Claude vision to transcribed note) and a "Paste Transcript" type (textarea + "Summarize with AI").
- **Theme**: light only (the warm cream design system). Dark mode has been removed.
- **Warmth**: per-contact temperature (unknown/cold/warm/hot/active), a colored dot on people cards and person sheets.

## Conventions worth knowing before editing

- **Custom dropdowns**: every categorical dropdown uses `SelectWithCustom` (drop-in `<select>` that appends "+ Add custom..."; picking it swaps to a text input). To persist custom values, pass `options={optionsWithCustom(DEFAULTS, customOptions, "field_name")}` and wrap `onChange` with `trackCustom("field_name", thatSameOptionsList, onAddCustomOption)`. `customOptions` (raw rows) and `onAddCustomOption` (`App`'s `addCustomOption`) are threaded down as props. Field names in use: institution_type, city, region, deal_stage, warmth, relationship, activity_type, priority, tag, strength, enabler_type. Multi-select tags use `TagPickerWithCustom`; colored button groups (warmth) use `ButtonGroupWithCustom`. All live near the top of `App.jsx`.
- Every "delete" CRUD function manually cascades related rows first (activities, junction tables, contact_roles, network_edges, todos) before deleting the parent, since there is no guaranteed `ON DELETE CASCADE`. `purgeDeal`/`purgeEnabler` archive a Target/Enabler while preserving shared history; `deleteInstitution` removes all backing rows.
- Colors for enum-like fields are defined once in `constants.js` as `{ id, label, color }` and rendered as `<span className="badge" style={{background: color+"22", color, border: \`1px solid ${color}44\`}}>`. Reuse this rather than inventing badge styles.
- Location fields use `SelectWithCustom` against `CITY_OPTIONS`/`REGION_OPTIONS`. Render a city as a small pin, `📍 {city}`, with the `.city-pin` class.
- Only send optional string fields to Supabase when they have real content (trim and skip empties); never send an empty string for a numeric column.
- Never use the em dash character anywhere: not in UI text, labels, toasts, descriptions, generated content, AI prompts, or code comments. Use commas, periods, colons, or parentheses instead. The AI prompts explicitly instruct the model to avoid them too.
- No test suite exists. Verify changes by running the dev server and exercising the feature directly. This app talks to a live, shared Supabase project with real data, so clean up any test data afterward, and do not assume a row is test data just because it appeared recently; verify before deleting.

## Mobile responsiveness

The app is responsive at `max-width: 768px` (phones); most of the mobile layout lives in one media block at the bottom of `src/styles.css`. Key behaviors: the left sidebar is hidden and replaced by a fixed bottom **tab bar** (`MobileTabBar` in `App.jsx`, rendered next to `CommentPanel`; **Home/Pipeline/Ecosystem/Tasks**, active tab in mango gold, `env(safe-area-inset-bottom)` for notches). The Network Map is **desktop-only**: it is dropped from the mobile tab bar, and if `view === "map"` on a phone the map renders a "Network Map is best viewed on desktop." notice (`.map-mobile-notice`) instead of the D3 graph. Reports and Notes have no mobile tab either; Reports is reached from the "View Reports" button at the bottom of Home, and Notes from a "My Notes" section on Home (the Notes list and editor are separate full-screen views on mobile, the editor gaining a back button). The floating comment button sits just above the tab bar, right-aligned (`right: 16px; bottom: calc(56px + env(safe-area-inset-bottom) + 12px)`), with the Ecosystem add FAB stacked above it; `.main` gets extra bottom padding so content scrolls clear of both. These two floating-button overrides live in a final media block at the very end of `styles.css` because the base `.comment-fab`/`.comment-panel` rules are defined after the main responsive block and would otherwise win the cascade. The Pipeline shows one full-width stage column at a time with CSS scroll-snap; `MobilePipelineNav` (a header with prev/next arrows over a `kanbanRef`) tracks and scrolls the centered column. The Ecosystem "+ Institution"/"+ Person" buttons collapse into a floating add button (`.mobile-add-fab` in `NetworkTab`); People forces card view (`useIsMobile()` -> `effectivePeopleView`) with a "Switch to desktop for table editing" note. Sheets/modals go full width (modals become full-screen sheets, inputs 16px to avoid iOS zoom), stats become a 2x2 grid, tap targets are >= 44px, and the Network Map side panel becomes a 60% bottom sheet with larger person node tap targets (min 16px radius) and `touch-action: none` for pinch/pan. The floating comment button moves above the tab bar (bottom-left, clear of the add FAB). `useIsMobile(768)` is a small `matchMedia` hook for the few JS-driven switches CSS cannot do.

## Voice notes (Quick Add)

Every `QuickAdd` (person and institution sheets, desktop and mobile) has a microphone button beside the activity-type selector. It records via the Web Speech API (`window.SpeechRecognition || webkitSpeechRecognition`, `lang: "en-US"`, `continuous: true`, `interimResults: false`); the raw transcription is accumulated in a ref and never shown. On stop, the raw text goes to `digestVoiceNote(transcript)` in `src/anthropic.js`, which returns `{ summary, action_items: [{title, priority, due_date_hint}] }` (JSON only, max_tokens 500, backticks stripped before parsing). The result renders as a "Meeting Note" card: an editable summary textarea plus a checklist of action items with priority badges. "Save Note" logs an activity of type `voice_note` (a 🎤 glyph in `ACTIVITY_GLYPHS`); "Save Note + Create Tasks" also creates todos linked to the sheet's entity via `onCreateTasks` (institution sheet uses its `onAddTodo`; person sheet gets an `onAddTodo` that stamps `contact_id`). `parseDueHint()` best-effort converts a due-date hint ("tomorrow", "next Thursday", "end of week") to an ISO date, otherwise the hint stays in the task title. If the browser lacks speech support, it shows "Voice notes not supported in this browser. Try Chrome or Safari."
